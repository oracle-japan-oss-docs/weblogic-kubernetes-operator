2019年6月6日 v2.2.0 FMW Infrastructureドメインのサポートが追加されました。 WebLogic Serverインスタンスはデフォルトで正常に停止され、停止オプションを構成できるようになりました。 オペレータがJDK 11で構築および実行されるようになりました。

2019年1月24日 v2.0 最終バージョン番号およびドキュメントの更新。

2018年12月18日 v2.0-rc1 Helmチャートを使用してオペレータがインストールされるようになり、以前のスクリプトが置き換えられました。 オペレータは、永続ボリュームまたはイメージのユースケースで、ドメイン・スキーマの再設計が必要なドメイン・ホームをサポートするようになりました。 ドメイン構成は、構成オーバーライド・テンプレートを使用してオーバーライドできます。 ロード・バランサおよびイングレスを個別に構成できるようになりました。 WebLogicログは、永続ボリュームまたはポッドのログに記録できます。 サーバーのライフ・サイクル・サポートが追加され、生成されたポッドの構成可能性が大幅に向上しました。 最終的なv2.0リリースは、オペレータ・チームが将来のリリースの一部として下位互換性を提供する予定の初期リリースになります。

主なテーマ

  • 補助イメージの簡略化:

    • ほとんどの場合、顧客は補助イメージの名前を指定するだけで済みます。

  • 新規クラスタ・リソース:

    • オペレータが、特定のWebLogicクラスタを構成し、そのクラスタのステータス情報を提供する新しいカスタム・リソース「クラスタ」を提供するようになりました。
    • このリソースは、「Kubernetes水平ポッド自動スケーリング(HPA)」などのテクノロジとともに使用して、WebLogicクラスタをスケーリングできます。

  • ステータスおよびイベントの更新:

    • ドメインおよびクラスタ・リソースについて生成された「条件」および「イベント」は、大幅に更新されました。
    • ドメインまたは個々のクラスタが、アプリケーション・ワークロードの目的の状態または重要な可用性に達すると、オペレータはより明確に通信します。
    • 失敗は、顧客の介入を必要とする失敗と、一時的な再試行が可能な失敗とで区別されます。

  • 更新を再試行:

    • 「失敗後の再試行」のオペレータ処理が大幅に更新されました。
    • お客様は、ドメインごとに再試行間隔および制限を調整したり、再試行を無効にできるようになりました。
    • ドメイン・リソース.statusから、最初の失敗、最後の再試行および次の再試行の予想時間を簡単に推測できるようになりました。

  • Istioの互換性:

    • ドメイン構成には、常にIstioおよびその他のサービス・メッシュ製品との互換性があります。
    • このサポートを有効にするために顧客が必要なくなりました。
アップグレードのノート

  • このリリースでは、ドメイン・リソースweblogic.oracle/v9の新しいAPIバージョンが導入されています。

    • オペレータ3.0.0で導入された以前のAPIバージョンweblogic.oracle/v8は非推奨ですが、引き続きサポートされています。

  • このリリースでは、新しいカスタム・リソースであるweblogic.oracle/v1クラスタ・リソースも導入されています。

    • このクラスタ・リソースは、特定のWebLogicクラスタを構成し、顧客はKubernetes Horizontal Pod Autoscaling (HPA)または同様のテクノロジを使用してそのクラスタをスケーリングできます。
    • クラスタ・リソースの.specスキーマは、以前のweblogic.oracle/v8ドメイン・リソースの.spec.clusters[*]コンテンツと類似し、置き換えられます。
    • クラスタ・リソースの.statusスキーマは、ドメイン・リソースの.status.clusters[*]コンテンツの内容をミラー化します。

  • このリリースでは、「オペレータのアップグレード」が簡略化されます。

    • オペレータに「スキーマ変換webフック」が追加されました。
      • このwebフックにより、weblogic.oracle/v8ドメイン・リソースのweblogic.oracle/v9へのアップグレードが自動化されます。
      • 顧客は最新のドメイン・スキーマの使用を開始するために何もアクションを実行する必要はありませんが、新しいプロジェクトについては、v9ドメイン・リソースから始めることをお薦めします。
    • Kubernetesクラスタごとに単一のWebLogic Kubernetes Operatorをインストールする顧客(最も一般的なユースケース)は、任意の3.xオペレータ・リリースから4.0.0に直接アップグレードできます。 4.0.0のHelmチャートでは、スキーマ変換webフックが自動的にインストールされます。
    • 単一のKubernetesクラスタに複数のWebLogic Kubernetes Operatorをインストールする顧客の場合:
      • オペレータを4.0.0にアップグレードする前に、すべてのオペレータを少なくともバージョン3.4.1にアップグレードする必要があります。
      • 4.0.0 Helmチャートでは、クラスタ内のすべての4.0.0オペレータで共有されるシングルトン・スキーマ変換webフックもインストールされるようになったため、webhookOnly Helmチャート・オプションを使用してインストールしますこのwebフックは、4.0.0オペレータをインストールする前に独自のネームスペースにあり、各オペレータでpreserveWebhook Helmチャート・オプションを使用して、オペレータのアンインストールによって共有webフックがアンインストールされないようにします。
    • このオペレータは、既存の"v8"ドメインYAMLファイルを"v9"に変換するために使用できるユーティリティを提供します。
    • 複数の「Helmチャートのデフォルト値が変更されました」。 Helmのアップグレード中に--reuse-valuesオプションを使用して3.xインストールをアップグレードする顧客は、元のインストールの値を引き続き使用します。
ドメイン・スキーマの変更

  • Model in Image関連補助イメージ。

    • 補助イメージの簡略化された構成のための.spec.configuration.model.auxiliaryImagesを追加しました。
    • .spec.configuration.model.auxiliaryImageVolumeMountPath、.spec.configuration.model.auxiliaryImageVolumeMedium、.spec.configuration.model.auxiliaryImageVolumeSizeLimitなど、補助イメージに関連する追加の高度な設定が追加されました。
    • 簡略化作業の一部として、.spec.serverPod.auxiliaryImagesおよび.spec.auxiliaryImageVolumesが削除されました。
    • spec.configuration.model.AuxiliaryImagesが指定されている場合はdomain.spec.configuration.model.wdtInstallHomeを/aux/weblogic-deployに、それ以外の場合は/u01/wdt/weblogic-deployにデフォルト設定します。 以前は、常に/u01/wdt/weblogic-deployにデフォルト設定されていました。
    • spec.configuration.model.AuxiliaryImagesが指定されている場合はdomain.spec.configuration.model.wdtModelHomeを/aux/modelsに、それ以外の場合は/u01/wdt/modelsにデフォルト設定します。 以前は、常に/u01/wdt/modelsにデフォルト設定されていました。

  • クラスタの変更。

    • .spec.clusters[*]の下のコンテンツを新しいクラスタ・リソース.specに移動しました。
    • .spec.clusters[*]がクラスタ・リソース名のリストになるように変更されました。
    • クラスタの最小値として削除されたallowReplicasBelowMinDynClusterSizeフィールドを.spec.および.spec.clusters[*]から自動スケーリングで構成する必要があります。

  • 再試行および失敗チューニング関連。

    • 失敗再試行のタイミングの構成に、.spec.failureRetryIntervalSecondsおよび.spec.failureRetryLimitMinutesが追加されました。
    • .spec.serverPod.maxReadyWaitTimeSecondsを追加して、起動中にサーバーが停止したと見なされるまでの最大時間を構成しました。

  • 条件ステータスが更新されます。

    • ドメインの現在の状態を明確に反映する条件を追加または更新しました:
      • Completed条件は、失敗が検出されなかった場合にTrueに設定され、予測されるすべてのポッドはready、ターゲットimage(s)、restartVersionおよびintrospectVersionに設定されます。
      • 十分な数のポッドがreadyの場合、Available条件はTrueに設定されます。 この条件は、CompletedがFalse、Failed条件がレポートされる場合、またはクラスタにreadyではない最大cluster.spec.maxUnavailableポッドがある場合でも、Trueにできます。
      • Failed条件がTrueに設定され、失敗発生時にその.messageフィールドが更新されます。
    • 失敗または警告を表す条件の重大度を説明する.status.conditions[*].severityが追加されました。
    • Failure条件がレポートされた時間の測定を可能にするフィールド.status.initialFailureTimeおよび.status.lastFailureTimeを追加しました。
    • 条件ServersReadyを削除しました。 かわりに、AvailableまたはCompletedを使用します。
    • 条件Progressingを削除しました。 かわりに、次のようにします。
      • CompletedがTrueであることを確認して、ドメインが完全に最新で稼働している時間を確認します。
      • AvailableまたはCompletedが存在するかどうかを確認して、オペレータがドメイン・リソースに気付いたことを確認します。
      • FailedがTrueの場合は、その.messageおよび.severityを確認して、再試行が停止したかどうかを確認します。

  • 一般ステータスの更新。

    • .metadata.generationとの比較のために.status.observedGenerationが追加されました。2つの値が一致する場合、ステータスはドメイン・リソースに対する最新の更新のステータスを正しく反映します。
    • .status.lastIntrospectJobProcessedUidの名前を.status.failedIntrospectionUidに変更します。
    • .status.serverStatus[*].podReadyおよび.status.serverStatus[*].podPhaseに、各WebLogic Serverインスタンスに関連付けられたポッドに関する追加情報が追加されました。

  • その他の変更。

    • WebLogic Serverログのディレクトリ・レイアウトを構成するための.spec.logHomeLayoutが追加されました。
    • サーバーの停止オプションの構成を拡張する.spec.serverPod.shutdown.waitForAllSessionsが追加されました。
    • このセクションは不要になったため、.spec.configuration.istioを削除しました。
    • 以前に非推奨になったフィールド.spec.domainHomeInImage、.spec.configOverridesおよび.spec.configOverrideSecretsを削除しました。 これらは、.spec.domainHomeSourceType、.spec.configuration.overridesConfigMapおよび.spec.configuration.secretsに置き換えられました。
    • このフィールドは不要になったため、serverStartStateフィールドを.spec, .spec.adminServer, .spec.managedServers[*]およびspec.clusters[*]から削除しました。
    • .spec.adminServer.adminChannelPortForwardingEnabledのデフォルトの変更はtrueです。 以前は、falseにデフォルト設定されていました。
