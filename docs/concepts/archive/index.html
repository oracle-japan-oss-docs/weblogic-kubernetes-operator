アーカイブ・ファイル

アーカイブ・ファイルは、バイナリおよびその他のファイル・リソースをターゲット・ドメインにデプロイするために使用されます。 アーカイブは、特定のディレクトリ構造を持つZIPファイルです。 ターゲット・システムに存在しないモデルで参照されるファイル・リソースは、アーカイブ内の正しいロケーションに格納する必要があり、モデルはアーカイブへのパスを反映する必要があります。 必要に応じて、モデル自体をアーカイブ内に格納することもできます。

モデルでターゲット・システム上の正しいロケーションが指定されている場合は、ターゲット・システムにすでに存在するファイル・リソースをアーカイブに含める必要はありません。

例

この例は、アーカイブに含まれるEARファイル・リソースを参照するSourcePath値を持つアプリケーションを示しています。

appDeployments:
    Application:
        simpleear :
            SourcePath: wlsdeploy/applications/simpleear.ear
            Target: 'my-cluster'
            ModuleType: ear

前述の例は、wlsdeploy/applications/simpleear.earの値を持つsimpleearアプリケーションの属性SourcePathを示しています。 プレフィクスwlsdeploy/は、リソースが指定されたロケーションのアーカイブ・ファイルにあり、ドメイン内のそのディレクトリ(この場合は<domain-home>/wlsdeploy/applications/simpleear.ear)にデプロイされることを示します。

アーカイブ構造

これらは、様々なタイプのリソースに使用されるアーカイブ内のパスです。 ユーザーは、これらのロケーションの下にさらにディレクトリ構造を作成して、ファイルおよびディレクトリを適宜編成できます。

atpwallet

Oracle Autonomous Transaction Processing Cloud Databaseで使用するためにウォレットを格納できるディレクトリ。 ファイル・リソース名はモデルで指定されておらず、このロケーションのアーカイブ内の単一のZIPファイルとみなされます。

model

オプションでモデルが配置されるディレクトリ。 YAMLまたはJSONのいずれかのモデル・ファイルのみが許可され、適切なYAMLまたはJSONファイル拡張子を持つ必要があります。

opsswallet

Oracle Platform Security Servicesで使用するためにウォレットを格納できるディレクトリ。 ファイル・リソース名はモデルで指定されておらず、このロケーションのアーカイブ内の単一のZIPファイルとみなされます。

wlsdeploy/applications

アプリケーションが格納されるルート・ディレクトリ。 アプリケーションは、EARまたはWARファイルとしてアーカイブに格納することも、このフォルダの下に展開することもできます。

拡張アプリケーション・ディレクトリは、WebLogic Deploy Toolingリリース1.6.2の後にサポートされます。

展開されたWARアプリケーションのサンプルには、次のエントリがあります:

wlsdeploy/applications/myApp/index.jsp
wlsdeploy/applications/myApp/META-INF/MANIFEST.MF
wlsdeploy/applications/myApp/WEB-INF/classes/MyClass.class
wlsdeploy/applications/myApp/WEB-INF/web.xml
wlsdeploy/applications/myApp/WEB-INF/weblogic.xml

wlsdeploy/classpathLibraries

サーバー・クラスパスに使用されるJAR/ディレクトリが格納されるルート・ディレクトリ。 モデルで参照されていないファイル・リソースも含め、このディレクトリの下のすべてのファイル・リソースが抽出されます。

wlsdeploy/coherence

Coherence永続ストアの空のディレクトリが存在する必要があるルート・ディレクトリ。

wlsdeploy/domainBin

$DOMAIN_HOME/binスクリプトが格納されるルート・ディレクトリ。 モデルのdomainInfo/domainBinセクションで参照されるスクリプトのみが抽出されます。

wlsdeploy/domainLibraries

$DOMAIN_HOME/libライブラリが格納されるルート・ディレクトリ。 モデルのdomainInfo/domainLibrariesセクションで参照されるライブラリのみが抽出されます。

wlsdeploy/nodeManager

キーストア・ファイルなどのノード・マネージャ・ファイル・リソースが格納されるルート・ディレクトリ。

wlsdeploy/scripts

スクリプトが格納されるルート・ディレクトリ。 これには、JDBC作成スクリプトおよびWLDFアクション・スクリプトが含まれます。

wlsdeploy/servers

キーストア・ファイルなどのサーバー・ファイルが格納されるルート・ディレクトリ。 これらは、wlsdeploy/server/my-server/mykey.jksなどのサーバー名で編成されています。

wlsdeploy/sharedLibraries

共有ライブラリが格納されるルート・ディレクトリ。 これらは、アーカイブ内にJARファイルとして格納されます。

wlsdeploy/stores

モデル内のFileStore要素に対して空のディレクトリが存在する必要があるルート・ディレクトリ。

複数のアーカイブ・ファイルの使用

ドメインの作成、ドメインの更新、アプリケーションのデプロイおよびモデルの検証ツールを使用すると、コマンドラインで複数のアーカイブ・ファイルを指定できます。 次に例を示します。 

$ weblogic-deploy\bin\createDomain.cmd -archive_file one.zip,two.zip,three.zip ...

ファイル・リソースは、これらのアーカイブのいずれにも存在できます。 各アーカイブのリソースは、以前のアーカイブで見つかったリソースより優先されます。

モデルが複数のアーカイブに存在するリソースを参照する場合、リスト内の最新のリソースが優先されます。 たとえば、モデルがwlsdeploy/applications/myapp.earを参照し、そのリソースがアーカイブone.zipおよびtwo.zipに存在する場合、two.zipのリソースが使用されます。

同様のルールは、想定ロケーションを持つが、モデル内で特に呼び出されないリソースに適用されます。 たとえば、アーカイブtwo.zipのウォレットがatpwallet/wallet2.zipにあり、three.zipのウォレットがatpwallet/wallet3.zipにある場合、ウォレットatpwallet/wallet3.zipが使用されます。

直接参照されずに抽出されたリソースは、archive_file引数で指定された順序でアーカイブから抽出されます。 たとえば、one.zipおよびtwo.zipのwlsdeploy/classpathLibrariesの下にリソースがある場合、one.zipのリソースは<domain-home>/wlsdeploy/classpathLibrariesに抽出され、two.zipのリソースは同じロケーションに抽出され、重複するファイルは上書きされます。
