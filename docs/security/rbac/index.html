概要

オペレータは、特定のKubernetesロールがKubernetesクラスタに作成されていることを前提としています。 オペレータHelmチャートでは、オペレータが使用するServiceAccountに必要なClusterRoles、ClusterRoleBindings、ロールおよびRoleBindingsが作成されます。 オペレータはまた、オペレータの実行開始時にRBAC設定が正しいことを確認しようとします。

オペレータが使用するKubernetes ServiceAccountの詳細は、「サービス・アカウント」を参照してください。

一般的な設計目標は、オペレータが必要とする最小限の権限をオペレータに付与し、Kubernetes組込みロールの使用に適したカスタム・ロールよりも組込みロールを優先することです。

Kubernetesロールの詳細は、Kubernetes RBACのドキュメントを参照してください。

オペレータRBAC定義

Helmリリース名weblogic-operatorを使用して、オペレータがインストールされたオペレータが使用したKubernetesロールおよび関連するバインディングを表示するには、Helm statusコマンドを使用するときにKubernetesオブジェクトRole, RoleBinding, ClusterRoleおよびClusterRoleBindingを探します:

$ helm status weblogic-operator

オペレータがdomain1-nsのターゲット・ネームスペースを持つネームスペースweblogic-operator-nsにインストールされている場合、次のコマンドを使用してKubernetesロールおよび関連RoleBindingsの「サブセット」を表示できます:

$ kubectl describe clusterrole \
  weblogic-operator-ns-weblogic-operator-clusterrole-general

$ kubectl describe clusterrolebinding \
  weblogic-operator-ns-weblogic-operator-clusterrolebinding-general

$ kubectl -n weblogic-operator-ns \
  describe role weblogic-operator-role

$ kubectl -n domain1-ns \
  describe rolebinding weblogic-operator-rolebinding-namespace \
KubernetesロールおよびRoleBindingのネーミング規則

RoleおよびRoleBindingオブジェクトには、次のネーミング・パターンが使用されます:

  • weblogic-operator-<type>-<optional-role-name>

次を使用します:

  1. Kubernetesオブジェクトの種類としての<type>:
    • role
    • rolebinding
  2. ロールまたはRoleBindingに指定されたオプション名としての<optional-role-name>
    • 次に例を示します: namespace

Kubernetes RoleBindingで作成されるオペレータの完全な名前は次のようになります:

weblogic-operator-rolebinding-namespace

Kubernetes ClusterRoleおよびClusterRoleBindingのネーミング規則

ClusterRoleおよびClusterRoleBindingオブジェクトには、次のネーミング・パターンが使用されます:

  • <operator-ns>-weblogic-operator-<type>-<role-name>

次を使用します:

  1. オペレータがインストールされているネームスペースとしての<operator-ns>
    • 次に例を示します: weblogic-operator-ns
  2. Kubernetesオブジェクトの種類としての<type>:
    • clusterrole
    • clusterrolebinding
  3. ロールまたはRoleBindingに指定された名前としての<role-name>
    • 次に例を示します: general

Kubernetes ClusterRoleBindingで作成されるオペレータの完全な名前は次のようになります:

weblogic-operator-ns-weblogic-operator-clusterrolebinding-general

RoleBindings

オペレータがKubernetes Namespace weblogic-operator-nsにインストールされ、オペレータのターゲット・ネームスペースがdomain1-nsである場合、次のRoleBindingエントリは、オペレータに権限を付与するRoleまたはClusterRoleにマップされます。

RoleBinding ロールにマップ済 リソース・アクセス ノート
weblogic-operator-rolebinding weblogic-operator-role 編集: シークレット、構成マップ、イベント RoleBindingは、weblogic-operator-ns 1というネームスペースに作成されます
weblogic-operator-rolebinding-namespace オペレータ・クラスタ・ロールnamespace Read: secrets, pods/log, pods/exec RoleBindingは、domain1-ns 2というネームスペースに作成されます
編集: configmaps, events, pods, services, jobs.batch
作成: pods/exec

ClusterRoleBindings

オペレータがKubernetes Namespace weblogic-operator-nsにインストールされている場合、次のClusterRoleBindingエントリは、オペレータに権限を付与するClusterRoleにマップされます。

ノート: 次の表のオペレータ名は、クラスタ名セクションの<role-name>を表します。

ClusterRoleBinding クラスタ・ロールにマップ済 リソース・アクセス ノート
Operator general Operator general Read: namespaces 3
編集: customresourcedefinitions
更新: ドメイン(weblogic.oracle)、ドメイン/ステータス
作成: tokenreview、selfsubjectrulesreviews
Operator nonresource Operator nonresource Get: /version/* 1
Operator discovery Kubernetes system:discovery 次を参照: Kubernetes検出ロール 3
Operator auth-delegator Kubernetes system:auth-delegator 次を参照: Kubernetesコンポーネント・ロール 3

  1. バインディングはオペレータServiceAccountに割り当てられます。↩︎

  2. バインディングは、オペレータが管理するように構成されている各ネームスペースのオペレータServiceAccountに割り当てられます。 「ドメイン・ネームスペースの管理」 ↩︎を参照

  3. バインディングはオペレータServiceAccountに割り当てられます。 また、オペレータのインストールで実際に設定されたKubernetes RBACリソースは、dedicated設定の値に基づいて調整されます。 デフォルトでは、dedicated値はfalseに設定されており、これらのセキュリティ・リソースはClusterRoleおよびClusterRoleBindingsとして作成されます。 dedicated値がtrueに設定されている場合、これらのリソースはオペレータのネームスペースにRolesおよびRoleBindingsとして作成されます。↩︎
