Oracle WebLogic Server Kubernetes Operator

Oracleは、WebLogic Serverを使用する組織が重要なワークロードを実行し、それらのワークロードをクラウドに移行する方法を見つけています。 DockerやKubernetesなどの業界標準に基づいて認証することで、WebLogicはクラウドに依存しないインフラストラクチャで実行されるようになりました。 また、Kubernetes環境でのWebLogicドメインのデプロイおよび管理を支援するいくつかの主要な機能を備えたオープン・ソースのOracle WebLogic Server Kubernetes Operator (“オペレータ”)も用意されています。 次のことが可能です。

  • Kubernetes PersistentVolumeでWebLogicドメインを作成します。 このPersistentVolumeは、NFSファイル・システムまたは他のKubernetesボリューム・タイプに配置できます。
  • コンテナ・イメージにWebLogicドメインを作成します。
  • WebLogicドメイン構成の特定の側面をオーバーライドします。
  • WebLogicドメインをKubernetesリソースとして定義します(Kubernetesカスタム・リソース定義を使用)。
  • 宣言的な起動パラメータおよび必要な状態に基づいてサーバーを起動します。
  • WebLogic構成済クラスタまたは動的クラスタを管理します。
  • 必要に応じて、WebLogic Server管理コンソールをKubernetesクラスタの外部に公開します。
  • 必要に応じて、Kubernetesドメインの外部でT3チャネルを公開します。
  • WebLogicドメイン内の管理対象サーバーが起動または停止されたときに、ロード・バランシングを使用してKubernetesドメイン外のWebLogicドメインでHTTPパスを公開し、ロード・バランサを更新します。
  • 管理対象サーバーをオンデマンドで起動および停止するか、WLDF、Prometheus、Grafanaまたはその他のルールに基づいてスケーリングを開始するためにREST APIと統合することで、WebLogicドメインをスケーリングします。
  • オペレータおよびWebLogic ServerのログをElasticsearchに公開し、Kibanaで対話します。

オペレータを最も速く体験するには、「クイックスタート・ガイド」をフォローするか、documentationを使用するか、「ブログ」を読むか、samplesを試します。

オペレータを使用して「チュートリアル」をステップ実行し、Oracle Cloud Infrastructure Container Engine for Kubernetes (OKE)クラスタにWebLogicドメイン・コンテナ・パッケージWebアプリケーションをデプロイして実行します。

現行本番リリース

「オペレータの現在のリリース」は3.1.3です。 このリリースは2021年2月12日に公開されました。 オペレータの前提条件およびサポートされている環境の「こちら」を参照してください。

オペレータの最近の変更および既知の問題については、「リリース・ノート」を参照してください。

オペレータの以前のバージョン

以前のリリースのオペレータのドキュメントは、「こちら」で入手できます。

下位互換性ガイドライン

2.0.1リリース以降、オペレータ・リリースは、ドメイン・リソース・スキーマ、オペレータHelmチャート入力値、構成オーバーライド・テンプレート、オペレータHelmチャートによって作成されたKubernetesリソース、オペレータによって作成されたKubernetesリソースおよびオペレータRESTインタフェースに関して下位互換性があります。 ここでは、3つのリリースの互換性を維持します。ただし、明確に伝達された非推奨の機能の場合を除きます。この機能は、代替機能が使用可能になった後も1つのリリースで維持されます。

このドキュメントについて

このドキュメントには、様々な読者を対象とした項が含まれています。 探しているものをより簡単に見つけるためには、次の目次を参照してください:

  • 「クイックスタート・ガイド」では、デフォルトを使用してオペレータを迅速に実行する方法が説明されており、特別なことはありません。
  • 「ユーザー・ガイド」には、オペレータのインストールおよび構成方法や、オペレータを使用してWebLogicドメインを作成および管理する方法など、詳細な使用方法の情報が含まれています。
  • 「サンプル」には、オペレータに関連する様々なタスクの実行方法を示す詳細なサンプル・コードと手順が用意されています。
  • 「開発者ガイド」では、オペレータの構築方法やテスト方法などを理解する必要があるユーザーの詳細が提供されます。 オペレータ・コードに貢献する必要があるユーザーは、ここで有用な情報を参照できます。 この項には、REST APIのAPIドキュメント(Javadoc)およびSwagger/OpenAPIドキュメントも含まれています。
  • 「寄与」セクションには、貢献要件に関する情報が表示されます。

ユーザー・ガイド

  • Kubernetesロード・バランサの構成。

  • ドメインの停止。
  • ドメインの削除。
  • 他にもたくさん!

オペレータへの貢献

Oracleは、誰からでもこのプロジェクトへの貢献を歓迎します。 貢献は、オペレータへの問題の報告や、プル・リクエストの発行でも結構です。 大量のプル・リクエストを引き起こす可能性のある重要な開発を開始する前に、まず問題を作成し、提案された変更について既存の開発者と話し合うことをお薦めします。

プル・リクエストを送信してバグを修正したり、既存の機能を強化する場合は、まず問題をオープンし、プル・リクエストを送信するときにその問題にリンクしてください。

送信の可能性について質問がある場合は、問題を自由にオープンできます。

Oracle WebLogic Server Kubernetes Operatorリポジトリへの貢献

プル・リクエストは、https://www.oracle.com/technetwork/community/oca-486395.htmlで入手可能なOracle Contributor Agreement (OCA)で行うことができます。

プル・リクエストを受け入れるには、OCA署名者リストに表示される貢献者の名前と電子メール・アドレスを使用して、コミット・メッセージの下部に次の行が必要です。

Signed-off-by: Your Name <you@example.org>

これは、次のものを使用してコミットすることで、プル・リクエストに自動的に追加できます:

git commit --signoff

OCAに署名したことを確認できるコミット者からのプル・リクエストのみを受け入れることができます。

プル・リクエスト・プロセス

  • リポジトリをフォークします。
  • フォークにブランチを作成して変更を実装します。 1234-fixesなどのブランチ名の一部として問題番号を使用することをお薦めします。
  • 修正に必要な変更でドキュメントが更新されていることを確認します。
  • ベース・イメージが変更された場合は、すべてのサンプルが更新されていることを確認します。
  • プル・リクエストを送信します。 プル・リクエストは空白のままにしないでください。 変更の目的を正確に説明し、変更の検証方法に関する簡単なステップを示します。 作成した問題も参照していることを確認します。 プル・リクエストをマージする前に、レビューのために2-3人に割り当てます。

新しい依存関係の導入

新しい依存関係を導入しようとするプル・リクエストは、追加のレビューの対象となることに注意してください。 通常、貢献者は互換性のないライセンスとの依存関係を回避し、最新バージョンの依存関係の使用を試みる必要があります。 新しい依存関係を受け入れる前に、標準のセキュリティ脆弱性チェックリストが参照されます。 WebLogic Serverを含むクローズド・ソース・コードへの依存性は、ほとんどの場合拒否されます。
