Domain home in image

サンプル・スクリプトは、WebLogic Image Tool (WIT)を使用したイメージでのWebLogicドメイン・ホームの作成を示しています。 サンプル・スクリプトには、WebLogicドメイン・ログ、サーバー・ログ、サーバー出力ファイルおよびノード・マネージャ・ログを既存のKubernetes PersistentVolume (PV)およびPersistentVolumeClaim (PVC)に配置するオプションがあります。 また、スクリプトはドメイン・リソースYAMLファイルを生成します。このファイルは、スクリプトで使用することも、WebLogic Serverポッドやサービスを含む対応するドメインのKubernetesアーティファクトを起動するために手動で使用することもできます。

前提条件

開始する前に、このドキュメント「ドメイン・リソース」をお読みください。

ドメイン作成スクリプトを実行する前に、次の前提条件を満たす必要があります:

  • WebLogic Image Toolでは、JAVA_HOMEがJava JDKバージョン8以降に設定されている必要があります。

  • オペレータには、パッチ29135930が適用されたOracle WebLogic Server 12.2.1.3.0またはOracle WebLogic Server 12.2.1.4.0またはOracle WebLogic Server 14.1.1.0.0のいずれかを含むWebLogicイメージが必要です。 既存のWebLogic Server一般提供イメージcontainer-registry.oracle.com/middleware/weblogic:12.2.1.3には、このパッチが適用されています。

    このサンプルでは、一般提供 (GA)イメージを使用しています。 GAイメージは、パブリック・インターネットから環境を使用できないデモンストレーションおよび開発目的「のみ」に適しています。「本番での使用はできません」。 本番では、常にOCRのCPU (パッチ適用済)イメージを使用するか、WebLogic Image Tool (WIT)と--recommendedPatchesオプションを使用してイメージを作成する必要があります。 詳細は、「Oracle WebLogic Serverの本番環境の保護」の「最新のパッチと更新の適用」を参照してください。

  • デフォルトのネームスペースを使用する場合を除き、ドメインのKubernetesネームスペースを作成します。

  • logHomeOnPVが有効になっている場合は、ログ・ホームをホストするKubernetes PersistentVolumeと、同じKubernetesネームスペース内のドメインのKubernetes PersistentVolumeClaimを作成します。 PVおよびPVCを作成するサンプルについては、「サンプルPVおよびPVCの作成」を参照してください。

  • フィールドusernameおよびpasswordを含むWebLogic管理者資格証明のKubernetesシークレットを作成し、シークレット名がweblogicCredentialsSecretNameに指定された値と一致することを確認します。次の「構成パラメータ」を参照してください。 例えば: 

$ cd ./kubernetes/samples/scripts/create-weblogic-domain-credentials

$ create-weblogic-credentials.sh
   -u <username>
   -p <password>
   -d domain1
   -n default
   -s domain1-weblogic-credentials

ノート: この例を使用して、weblogicCredentialsSecretNameをdomain1-weblogic-credentialsとして構成します。

スクリプトを使用してドメインを作成

create-domain.shスクリプトは、新しいドメイン・ホームとその中の異なる内部domain secretを使用して、実行ごとに新しいコンテナ・イメージを生成します。 同じドメイン内に異なるドメイン・シークレットを持つ異なるイメージが存在しないようにするには、新しいドメインでアクティブなドメインとは異なるdomainUIDを使用するか、次のコマンドを使用して既存のドメインを削除し、すべてのWebLogic Serverインスタンスポッドが終了するまで待機してから、同じdomainUIDを持つドメインを作成することを強くお薦めします: $ kubectl delete domain [domainUID] -n [domainNamespace]

ドメインを作成するサンプルは、次のディレクトリにあります:

$ cd kubernetes/samples/scripts/create-weblogic-domain/domain-home-in-image

create-domain-inputs.yamlファイルのコピーを作成し、正しい値で更新し、作成スクリプトを実行して、入力ファイルと出力ディレクトリを指し、WebLogic管理者のユーザー名とパスワードとともに作成します:

$ ./create-domain.sh \
  -u <username> \
  -p <password> \
  -i create-domain-inputs.yaml \
  -o /<path to output-directory>

create-domain.shスクリプトとその入力ファイルは、デモンストレーション目的「のみ」用です。その内容および生成するドメイン・リソース・ファイルは、予告なしに変更される場合があります。 本番環境では、WebLogic Image ToolおよびWebLogic Deploy Tooling (適用される場合)、およびドメイン・リソース・ファイルを直接使用することを強くお勧めします。

このスクリプトは、次のステップを実行します:

  • このドメイン用に生成されたプロパティおよびKubernetes YAMLファイルのディレクトリを作成します(まだ存在しない場合)。 パス名は/<path to output-directory>/weblogic-domains/<domainUID>です。 ディレクトリがすでに存在する場合は、その内容が削除されます。

  • 前述の手順で作成したディレクトリにプロパティ・ファイルdomain.propertiesを作成します。 このプロパティ・ファイルは、サンプルWebLogic Serverドメインを作成するために使用されます。 スクリプトが正常に完了すると、domain.propertiesファイルが削除されます。

  • 最新のWebLogic Deploy Tooling (WDT)およびWebLogic Image ToolインストーラのZIPファイルを/tmp/dhii-sample/toolsディレクトリにダウンロードします。 Domain in Imageコンテナ・イメージを作成するにはWITが必要です。wdtモードを使用する場合はWDTが必要です。 各リリースの最新バージョンを確認するには、GitHub 「WebLogic Deploy Toolリリース」および「WebLogic Image Toolリリース」 webページにアクセスします。

  • <toolsDir>/imagetoolディレクトリにWebLogic Image Toolを設定します。<toolsDir>は、入力YAMLファイルのtoolsDirパラメータで指定されたディレクトリです。 WITキャッシュ・ストアのロケーションを<tools>/imagetool-cacheディレクトリに設定し、ツールのキャッシュにwdt_<WDT_VERSION>エントリを配置します。これは、WDT ZIPファイル・インストーラのパスを指します。 WITキャッシュの詳細については、「WITキャッシュのドキュメント」を参照してください。

  • オプションの-nオプションおよび暗号化キーが指定されている場合は、入力ファイルのdomainHomeImageBaseパラメータで指定されたイメージを実行しているコンテナでWDT 「モデルの暗号化ツール」を起動し、domain.propertiesファイルのパスワード・プロパティを暗号化します。 このパスワードは、入力YAMLファイルのmodeパラメータの値がwlstである場合にスキップされます。これは、この機能がWDTによって提供されるためです。

  • WebLogic Image Toolを起動し、入力ファイルからdomainHomeImageBaseパラメータで指定されたWebLogicイメージに基づいて新しいWebLogic Serverドメインを作成します。 新しいWebLogic Serverドメインは、入力YAMLファイルのmodeパラメータの値に基づいて、次のいずれかのオプションを使用して作成されます:

    • modeパラメータの値がwdtの場合、createDomainWdtModelパラメータで指定されたWDTモデルおよびdomain.propertiesファイル内のWDT変数は、WebLogic Image Toolによって新しいWebLogic Serverドメインが作成されます。
    • modeパラメータの値がwlstの場合、createDomainWlstScriptパラメータで指定されたオフラインWLSTスクリプトが実行され、新しいWebLogic Serverドメインが作成されます。

  • 生成されたイメージは、入力ファイルに指定されたimageパラメータでタグ付けされます。

    Oracleは、ドメイン・ホームを含むイメージをプライベートとしてレジストリ(Oracle Cloud Infrastructure Registry、GitHub Container Registryなど)に格納することを強くお薦めします。これは、このイメージには、外部リソースへのアクセスに使用されるキーや資格証明(データ・ソース・パスワードなど)など、ドメインに関する機密情報が含まれるためです。 詳細は、「コンテナ・イメージ保護」を参照してください。

  • Kubernetesドメイン・リソースYAMLファイルdomain.yamlを、前述のディレクトリ内に作成します。 このYAMLファイルは、kubectl create -fまたはkubectl apply -fコマンドを使用してKubernetesリソースを作成するために使用できます。 

$ kubectl apply -f /<path to output-directory>/weblogic-domains/<domainUID>/domain.yaml

便宜上、スクリプトでは、-eオプションを使用して、オプションでドメイン・オブジェクトを作成できます。これにより、対応するWebLogic Serverポッドおよびサービスが作成されます。 このオプションは、単一ノードのKubernetesクラスタでのみ使用する必要があります。

マルチ・ノードKubernetesクラスタの場合は、kubectl apply -fコマンドを使用してドメイン・リソースYAMLファイルを作成する前に、生成されたイメージがすべてのノードで使用可能であることを確認してください。

作成スクリプトの使用方法は次のとおりです:

$ sh create-domain.sh -h

usage: create-domain.sh -o dir -i file -u username -p password [-n encryption-key] [-e] [-v] [-h]
  -i Parameter inputs file, must be specified.
  -o Output directory for the generated properties and YAML files, must be specified.
  -u WebLogic administrator user name for the WebLogic domain.
  -p WebLogic administrator Password for the WebLogic domain.
  -e Also create the resources in the generated YAML files, optional.
  -v Validate the existence of persistentVolumeClaim, optional.
  -n Encryption key for encrypting passwords in the WDT model and properties files, optional.
  -h Help

サンプル・スクリプトを別のロケーションにコピーする場合は、<weblogic-kubernetes-operator-project>/kubernetes/samples/scriptsディレクトリ内のすべてをターゲット・ディレクトリにコピーし、元のディレクトリ階層を維持してください。

スクリプトによって作成されるデフォルト・ドメインには、次の特性があります:

  • ポート7001でリスニングするadmin-serverという名前の管理サーバー。
  • サイズ5のcluster-1という名前の動的クラスタ。
  • ポート8001でリスニングする、managed-server1およびmanaged-server2という名前の2つの管理対象サーバー。
  • アプリケーションはデプロイされていません
  • T3チャネル。

Kubernetesクラスタに対してリモートのマシンからサンプルを実行し、クラスタに対してローカルなレジストリに新しいイメージをプッシュする必要がある場合は、次の手順を実行する必要があります(「構成パラメータ」表のimageプロパティも参照。):

  • 入力ファイルのimageプロパティをターゲット・イメージ名(必要に応じて、レジストリ・ホスト名、ポートおよびタグを含む)に設定します。
  • Kubernetesでプライベート・レジストリからイメージをプルする場合は、資格証明を保持するKubernetesシークレットを作成し、入力ファイルのimagePullSecretNameプロパティをシークレットの名前に設定します。

    Kubernetesのシークレットは、ドメインが実行されるネームスペースと同じネームスペースに存在する必要があります。 詳細は、「コンテナ・イメージ保護」を参照してください。

  • -eオプションを指定せずにcreate-domain.shスクリプトを実行します。
  • imageをターゲット・レジストリにプッシュします。
  • 次のコマンドを実行してドメインを作成します:
$ kubectl apply -f /<path to output-directory>/weblogic-domains/<domainUID>/domain.yaml

ドメイン作成の入力は、create-domain-inputs.yamlを編集してカスタマイズできます。

構成パラメータ

次のパラメータを入力ファイルに指定できます。

パラメータ 定義 デフォルト
sslEnabled 各WebLogic Serverインスタンスに対してSSLを有効にするかどうかを示すブール値。 false
adminPort Kubernetesクラスタ内の管理サーバーのポート番号。 7001
adminServerSSLPort Kubernetesクラスタ内の管理サーバーのSSLポート番号。 7002
adminNodePort Kubernetesクラスタ外の管理サーバーのポート番号。 30701
adminServerName 管理サーバーの名前。 admin-server
clusterName ドメインに対して生成するWebLogicクラスタ・インスタンスの名前。 cluster-1
configuredManagedServerCount ドメインに対して生成する管理対象サーバー・インスタンスの数。 5
createDomainWdtModel WDT modeを使用するときに、ドメインの作成スクリプトがWebLogicドメインの作成に使用するWDTモデルYAMLファイル。 modeがwlstに設定されている場合、この値は無視されます。 wdt/wdt_model_dynamic.yaml
createDomainWlstScript WLST modeを使用する場合、ドメインの作成スクリプトでWebLogicドメインの作成に使用するWLSTスクリプト。 modeがwdt (デフォルトのmode)に設定されている場合、この値は無視されます。 wlst/create-wls-domain.py
domainHome 生成されたWebLogic Serverイメージに作成されるWebLogicドメインのドメイン・ホーム・ディレクトリ。 /u01/oracle/user_projects/domains/<domainUID>
domainHomeImageBase WebLogicドメイン・イメージの構築に使用されるベースWebLogicバイナリ・イメージ。 オペレータには、パッチ29135930が適用されたOracle WebLogic Server 12.2.1.3.0、Oracle WebLogic Server 12.2.1.4.0またはOracle WebLogic Server 14.1.1.0.0が必要です。 既存のWebLogic Server一般提供 (GA)イメージcontainer-registry.oracle.com/middleware/weblogic:12.2.1.3には、このパッチが適用されています。 ノート: 基本イメージの例は、デモおよび開発目的「のみ」に適したGAイメージで、パブリック・インターネットから環境を使用できません。これらは「本番で使用できません」。 本番では、常にOCRのCPU (パッチ適用済)イメージを使用するか、WebLogic Image Tool (WIT)と--recommendedPatchesオプションを使用してイメージを作成する必要があります。 イメージの取得または作成方法の詳細は、「WebLogicイメージ」を参照してください。 container-registry.oracle.com/middleware/weblogic:12.2.1.3
domainPVMountPath ドメイン永続ボリュームのマウント・パス。 logHomeOnPVがtrueの場合、このパラメータは必須です。 それ以外の場合、このパラメータは無視されます。 /shared
domainUID この特定のドメインの識別に使用される一意のID。 生成されたWebLogicドメインの名前およびドメインの名前として使用されます。 このIDは、Kubernetesクラスタ内のすべてのドメインで一意である必要があります。 このIDには、Kubernetesサービス名に無効な文字を含めることはできません。 domain1
exposeAdminNodePort 管理サーバーがKubernetesクラスタの外部に公開されているかどうかを示すブール値。 false
exposeAdminT3Channel T3管理チャネルがKubernetesクラスタの外部に公開されているかどうかを示すブール値。 false
httpAccessLogInLogHome logHomeOnPVがtrueの場合に、サーバーのHTTPアクセス・ログ・ファイルをlogHomeと同じディレクトリに書き込むかどうかを示すブール値。 それ以外の場合、サーバーのHTTPアクセス・ログ・ファイルは、WebLogicドメイン・ホーム構成で指定されたディレクトリに書き込まれます。 true
image オペレータがドメインの起動に使用するWebLogic Serverイメージ。 ドメイン作成スクリプトは、ドメイン・ホームを含むWebLogic Serverイメージを生成します。 デフォルトでは、スクリプトは生成されたWebLogic Serverイメージにdomain-home-in-imageというタグを付け、それとdomainHomeImageBaseから取得したタグを使用して、生成されたドメイン・リソースYAMLファイルでimage要素を設定します。 このプロパティが設定されている場合、ドメインの作成スクリプトは、デフォルト値の代わりに指定された値を使用して、生成されたイメージにタグ付けし、ドメイン・リソースYAMLファイルでimageを設定します。 スクリプトを使用して作成されるドメインごとに一意の値が必要です。 ドメインを実行するKubernetesクラスタに対してリモートのマシンからサンプル・スクリプトを実行する場合は、このプロパティを、そのKubernetesクラスタに対してローカルなレジストリで使用するイメージ名に設定する必要があります。 また、kubectl create -fまたはkubectl apply -fコマンドを使用してドメインを起動する前に、imageをそのレジストリにプッシュする必要があります。 domain-home-in-image:<tag from domainHomeImageBase>
imagePullPolicy WebLogic Serverイメージ・プル・ポリシー。 有効な値は、IfNotPresent、AlwaysまたはNeverです。 IfNotPresent
imagePullSecretName WebLogic Serverイメージをプルするためにコンテナ・レジストリにアクセスするKubernetesシークレットの名前。 このパラメータを指定すると、シークレットの存在が検証されます。
includeServerOutInPodLog ポッドのstdoutにサーバー.out intを含めるかどうかを示すブール値。 true
initialManagedServerReplicas ドメインに対して最初に起動する管理対象サーバーの数。 2
javaOptions 管理サーバーおよび管理対象サーバーを起動するためのJavaオプション。 Javaオプションには、WebLogicドメイン情報を取得するために、次の事前定義済変数のいずれかを参照できます: $(DOMAIN_NAME), $(DOMAIN_HOME), $(ADMIN_NAME), $(ADMIN_PORT)、および$(SERVER_NAME)。 sslEnabledがtrueに設定され、WebLogicデモ証明書が使用されている場合は、-Dweblogic.security.SSL.ignoreHostnameVerification=trueを追加して、起動時に管理対象サーバーが管理サーバーに接続できるようにします。 この環境で生成されたWebLogicデモ証明書には、通常、ランタイム・コンテナのホスト名とは異なるホスト名が含まれています。 -Dweblogic.StdoutDebugEnabled=false
logHomeOnPV ログ・ホームを永続ボリュームに格納するかどうかを指定します。 trueに設定した場合、logHome、persistentVolumeClaimNameおよびdomainPVMountPathパラメータを指定する必要があります。 false
logHome ドメイン・ログ、サーバー・ログ、サーバー出力、ノード・マネージャ・ログ、イントロスペクタ出力およびサーバーHTTPアクセス・ログ・ファイルのインポッド・ロケーション。 指定しない場合、値はdomainUIDから/shared/logs/<domainUID>として導出されます。 logHomeOnPVがtrueの場合、このパラメータは必須です。 それ以外の場合、このパラメータは無視されます。 /shared/logs/domain1
managedServerNameBase 管理対象サーバー名の生成に使用されるベース文字列。 managed-server
managedServerPort 各管理対象サーバーのポート番号。 8001
managedServerSSLPort 各管理対象サーバーのSSLポート番号。 8002
mode createDomainWdtModelで指定されたWDTモデルを使用するか、createDomainWlstScriptで指定されたオフラインWLSTスクリプトを使用してWebLogicドメインを作成するか。 有効な値は、wdtまたはwlstです。 wdt
namespace ドメインを作成するKubernetesネームスペース。 default
persistentVolumeClaimName 永続ボリューム要求の名前。 指定しない場合、値はdomainUIDから<domainUID>-weblogic-sample-pvcとして導出されます。 logHomeOnPVがtrueの場合、このパラメータは必須です。 それ以外の場合、このパラメータは無視されます。 domain1-weblogic-sample-pvc
productionModeEnabled 本番モードがドメインに対して有効かどうかを示すブール値。 true
serverStartPolicy 起動するWebLogic Serverインスタンスを決定します。 有効な値はNEVER, IF_NEEDED, ADMIN_ONLYです。 IF_NEEDED
t3ChannelPort ネットワーク・アクセス・ポイントのT3チャネルのポート。 30012
t3PublicAddress T3チャネルのパブリック・アドレス。 これはKubernetesクラスタのパブリック・アドレスに設定する必要があります。 通常、これはロード・バランサ・アドレスです。 開発環境の場合のみ、単一サーバー(オール・イン・ワン)のKubernetesデプロイメントでは、これをマスターのアドレスに設定することも、少なくともワーカー・ノードのいずれかのアドレスに設定することもできます。 指定しない場合、スクリプトはそれをKubernetesクラスタのIPアドレスに設定しようとします。
weblogicCredentialsSecretName 管理サーバーのユーザー名およびパスワードのKubernetesシークレットの名前。 domain1-weblogic-credentials
serverPodCpuRequest, serverPodMemoryRequest, serverPodCpuCLimit, serverPodMemoryLimit 各サーバーポッドに許可されるコンピュート・リソースの最大量、および必要なコンピュート・リソースの最小量。 詳細は、Managing Compute Resources for ContainersのKubernetesのドキュメントを参照してください。 リソース・リクエストおよびリソース制限が指定されていません。
toolsDir WebLogic Deploy ToolおよびWebLogic Image Toolがインストールされているディレクトリ。 これらのツールがまだインストールされていない場合、スクリプトはこのディレクトリにインストールされます。 /tmp/dhii-sample/tools
wdtVersion スクリプトによってインストールされるWebLogic Deploy Toolのバージョン。 これは、1.9.10やLATESTなどの特定のバージョンにできます。 LATEST
witVersion スクリプトによってインストールされるWebLogic Image Toolのバージョン。 これは、1.9.10やLATESTなどの特定のバージョンにできます。 LATEST

生成されたYAMLファイル内のKubernetesリソースの名前は、入力YAMLファイルに指定されたプロパティの一部の値を使用して形成される可能性があることに注意してください。 これらのプロパティには、adminServerName、clusterNameおよびmanagedServerNameBaseが含まれます。 これらの値にKubernetesサービス名に無効な文字が含まれている場合、それらの文字は生成されたYAMLファイルで有効な値に変換されます。 たとえば、大文字は小文字に変換され、アンダースコア("_")はハイフン("-")に変換されます。

このサンプルでは、単一のクラスタのみを持つドメインのWebLogicドメイン・ホームおよび関連するKubernetesリソースを作成する方法を示します。 また、このサンプルでは、ユーザーが独自のスクリプトを指定して他のユースケースのドメイン・ホームを作成する機能も提供しています。 また、生成されたドメイン・リソースYAMLファイルを変更して、より多くのユースケースに対応できます。

結果の検証

作成スクリプトは、ドメインが作成されたことを確認し、エラーが発生した場合は失敗を報告します。 ただし、スクリプトによって作成された様々なKubernetesオブジェクトを理解するだけの場合でも、ドメインを手動で検証することが望ましい場合があります。

ドメインの削除

/<path to output-directory>/weblogic-domains/<domainUID>ディレクトリに生成されたYAMLファイルを使用して、Kubernetesリソースを削除できます。 次のコマンドを使用して、ドメインを削除します: 

$ kubectl delete -f domain.yaml

生成されたイメージの削除

不要になったら、生成されたイメージを削除します。 イメージがローカル・リポジトリにある場合は、次のコマンドを使用して、domain-home-in-image:12.2.1.4でタグ付けされたイメージを削除します : 

$ docker rmi domain-home-in-image:12.2.1.4

ツール・ディレクトリの削除

不要になったら、WebLogic Deploy ToolおよびWebLogic Image Toolがインストールされているディレクトリを削除します。 デフォルトでは、/tmp/dhii-sample/toolsディレクトリにインストールされます。 

$ rm -rf /tmp/dhii-sample/tools/
