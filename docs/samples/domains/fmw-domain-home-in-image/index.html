FMWインフラストラクチャdomain home in image

サンプル・スクリプトは、WebLogic Image Tool (WIT)を使用したイメージでのFMW Infrastructureドメイン・ホームの作成を示しています。 サンプル・スクリプトには、WebLogicドメイン・ログ、サーバー・ログ、サーバー出力ファイルおよびノード・マネージャ・ログを既存のKubernetes PersistentVolume (PV)およびPersistentVolumeClaim (PVC)に配置するオプションがあります。 また、スクリプトはドメイン・リソースYAMLファイルを生成します。このファイルは、スクリプトで使用することも、WebLogic Serverポッドやサービスを含む対応するドメインのKubernetesアーティファクトを起動するために手動で使用することもできます。

Domain in Image 「ドメイン・ホーム・ソース・タイプ」は、WebLogic Kubernetes Operatorバージョン4.0では非推奨です。 Oracleでは、必要に応じてDomain in PVまたはModel in Imageを選択することをお薦めします。

前提条件

開始する前に、このドキュメント「ドメイン・リソース」をお読みください。

ドメイン作成スクリプトを実行する前に、次の前提条件を満たす必要があります:

  • WebLogic Kubernetes Operatorが実行されていることを確認します。
  • オペレータには、パッチ29135930が適用されたFMW Infrastructure 12.2.1.3.0またはFMW Infrastructure 12.2.1.4.0のいずれかを含むイメージが必要です。 イメージを取得または作成する方法の詳細は、「FMW Infrastructureドメイン」を参照してください。 ノート: 2022年12月現在、Fusion Middleware 12.2.1.3はサポートされていません。 FMW Infrastructure 12.2.1.3の最後のCritical Patch Updates (CPU)イメージは、2022年10月に公開されました。
  • デフォルトのネームスペースを使用する場合を除き、ドメインのKubernetesネームスペースを作成します。
  • logHomeOnPVが有効になっている場合は、ログ・ホームをホストするKubernetes PersistentVolumeと、同じKubernetesネームスペース内のドメインのKubernetes PersistentVolumeClaimを作成します。 PVおよびPVCを作成するサンプルについては、「サンプルPVおよびPVCの作成」を参照してください。
  • ドメインと同じKubernetesネームスペースに管理アカウントのKubernetesシークレットusernameおよびpasswordを作成します。
  • 制限付きJRFドメインを作成しないかぎり、次のことも必要です:
    • データベースへのアクセスを構成します。 詳細は、「こちら」を参照してください。
    • RCU資格証明を使用してKubernetesシークレットを作成します。 詳細は、この「ドキュメント」を参照してください。

スクリプトを使用してドメインを作成

ドメインを作成するサンプルは、次のディレクトリにあります:

$ cd kubernetes/samples/scripts/create-fmw-infrastructure-domain/domain-home-in-image

create-domain-inputs.yamlファイルのコピーを作成し、正しい値で更新します。 fwmDomainTypeがJRFの場合は、rcuSchemaPrefix, rcuSchemaPassword, rcuDatabaseURL、rcuCredentialSecretsなど、RCUデータベース・スキーマにアクセスするための構成で入力ファイルを更新します。 作成スクリプトを実行し、入力ファイルと出力ディレクトリを指し、WebLogic管理者のユーザー名とパスワードとともに、JFR FMWドメインを作成する場合はRCUスキーマのパスワードも指定します: 

$ ./create-domain.sh \
  -u <username> \
  -p <password> \
  -q <RCU schema password> \
  -i create-domain-inputs.yaml \
  -o /<path to output-directory>

create-domain.shスクリプトとその入力ファイルは、デモンストレーション目的「のみ」用です。その内容および生成するドメイン・リソース・ファイルは、予告なしに変更される場合があります。 本番環境では、WebLogic Image ToolおよびWebLogic Deploy Tooling (適用される場合)、およびドメイン・リソース・ファイルを直接使用することを強くお勧めします。

このスクリプトは、次のステップを実行します:

  • このドメイン用に生成されたKubernetes YAMLファイルのディレクトリを作成します(まだ存在しない場合)。 パス名は/<path to output-directory>/weblogic-domains/<domainUID>です。 ディレクトリがすでに存在する場合は、このスクリプトを使用する前にその内容を削除する必要があります。

  • 以前に作成したディレクトリにプロパティ・ファイルdomain.propertiesを作成します。 このプロパティ・ファイルは、サンプルのFMWインフラストラクチャ・ドメインの作成に使用されます。 スクリプトが正常に完了すると、domain.propertiesファイルが削除されます。

  • 最新のWebLogic Deploy Tooling (WDT)およびWebLogic Image ToolインストーラのZIPファイルを/tmp/dhii-sample/toolsディレクトリにダウンロードします。 Domain in Imageコンテナ・イメージを作成するにはWITが必要です。wdtモードを使用する場合はWDTが必要です。 各リリースの最新バージョンを確認するには、GitHub 「WebLogic Deploy Toolリリース」および「WebLogic Image Toolリリース」 webページにアクセスします。

  • <toolsDir>/imagetoolディレクトリにWebLogic Image Toolを設定します。<toolsDir>は、入力YAMLファイルのtoolsDirパラメータで指定されたディレクトリです。 WITキャッシュ・ストアのロケーションを<tools>/imagetool-cacheディレクトリに設定し、ツールのキャッシュにwdt_<WDT_VERSION>エントリを配置します。これは、WDT ZIPファイル・インストーラのパスを指します。 WITキャッシュの詳細については、「WITキャッシュのドキュメント」を参照してください。

  • オプションの-nオプションおよび暗号化キーが指定されている場合は、入力ファイルのdomainHomeImageBaseパラメータで指定されたイメージを実行しているコンテナでWDT 「モデルの暗号化ツール」を起動し、domain.propertiesファイルのパスワード・プロパティを暗号化します。 このパスワードは、入力YAMLファイルのmodeパラメータの値がwlstである場合にスキップされます。これは、この機能がWDTによって提供されるためです。

  • WebLogic Image Toolを起動して、入力ファイルのdomainHomeImageBaseパラメータで指定されたFMWインフラストラクチャ・イメージに基づいて新しいFWMインフラストラクチャ・ドメインを作成します。 新しいWebLogic Serverドメインは、入力YAMLファイルのmodeパラメータの値に基づいて、次のいずれかのオプションを使用して作成されます:

    • modeパラメータの値がwdtの場合、createDomainWdtModelパラメータで指定されたWDTモデルおよびdomain.propertiesファイル内のWDT変数は、WebLogic Image Toolによって新しいWebLogic Serverドメインが作成されます。
    • modeパラメータの値がwlstの場合、createDomainWlstScriptパラメータで指定されたオフラインWLSTスクリプトが実行され、新しいWebLogic Serverドメインが作成されます。

  • 生成されたイメージは、入力ファイルに指定されたimageパラメータでタグ付けされます。

    Oracleは、ドメイン・ホームを含むイメージをプライベートとしてレジストリ(Oracle Cloud Infrastructure Registry、GitHub Container Registryなど)に格納することを強くお薦めします。これは、このイメージには、外部リソースへのアクセスに使用されるキーや資格証明(データ・ソース・パスワードなど)など、ドメインに関する機密情報が含まれるためです。 詳細は、「コンテナ・イメージ保護」を参照してください。

  • 以前に作成したディレクトリに、Kubernetesドメイン・リソースのYAMLファイルdomain.yamlを作成します。 このYAMLファイルは、kubectl create -fまたはkubectl apply -fコマンドを使用してKubernetesリソースを作成するために使用できます: 

    $ kubectl apply -f /<path to output-directory>/weblogic-domains/<domainUID>/domain.yaml

便宜上、スクリプトでは、-eオプションを使用して、オプションでドメイン・オブジェクトを作成できます。これにより、対応するWebLogic Serverポッドおよびサービスも作成されます。

作成スクリプトの使用方法は次のとおりです:

$ sh create-domain.sh -h

usage: create-domain.sh -o dir -i file -u username -p password [-q rcuSchemaPassword] [-b buildNetworkParam] [-n encryption-key] [-e] [-v] [-h]
  -i Parameter inputs file, must be specified.
  -o Output directory for the generated YAML files, must be specified.
  -u WebLogic administrator user name for the WebLogic domain.
  -p WebLogic administrator Password for the WebLogic domain.
  -q Password for the RCU schema. Required for JRF FMW domain type.
  -e Also create the resources in the generated YAML files, optional.
  -v Validate the existence of persistentVolumeClaim, optional.
  -n Encryption key for encrypting passwords in the WDT model and properties files, optional.
  -b Value to be used in the buildNetwork parameter when invoking WebLogic Image Tool, optional.
  -h Help

サンプル・スクリプトを別のロケーションにコピーする場合は、<weblogic-kubernetes-operator-project>/kubernetes/samples/scriptsディレクトリ内のすべてをターゲット・ディレクトリにコピーし、元のディレクトリ階層を維持してください。

スクリプトによって作成されるデフォルト・ドメインには、次の特性があります:

  • ポート7001でリスニングするadmin-serverという名前の管理サーバー。
  • サイズ3のcluster-1という構成済クラスタ。
  • managed-server1、managed-server2などの3つの管理対象サーバーは、ポート8001でリスニングします。
  • /shared/logs/<domainUID>にあるログ・ファイル。
  • アプリケーションはデプロイされていません
  • データ・ソースまたはJMSリソースがありません。
  • T3チャネル。

ドメイン作成の入力は、create-domain-inputs.yamlを編集してカスタマイズできます。

exposeAdminT3Channel T3管理チャネルがKubernetesクラスタの外部に公開されているかどうかを示すブール値。 false
fmwDomainType FMWインフラストラクチャ・ドメイン・タイプ。 有効な値は、JRFまたはRestrictedJRFです。 JRF
httpAccessLogInLogHome サーバーのHTTPアクセス・ログ・ファイルをlogHomeと同じディレクトリに書き込むかどうかを示すブール値。 それ以外の場合、サーバーのHTTPアクセス・ログ・ファイルは、WebLogicドメイン・ホーム構成で指定されたディレクトリに書き込まれます。 true

imagePullPolicy Oracle FMW Infrastructure Serverイメージ・プル・ポリシー。 有効な値は、IfNotPresent、AlwaysまたはNeverです。 IfNotPresent
imagePullSecretName WebLogic Serverイメージをプルするためにコンテナ・レジストリにアクセスするKubernetesシークレットの名前。 このパラメータを指定すると、シークレットの存在が検証されます。
includeServerOutInPodLog ポッドのstdoutにサーバー.outを含めるかどうかを示すブール値。 true
initialManagedServerReplicas ドメインに対して最初に起動する管理対象サーバーの数。 1
javaOptions 管理サーバーおよび管理対象サーバーを起動するためのJavaオプション。 Javaオプションには、WebLogicドメイン情報を取得するために、次の事前定義済変数のいずれかを参照できます: $(DOMAIN_NAME), $(DOMAIN_HOME), $(ADMIN_NAME), $(ADMIN_PORT)、および$(SERVER_NAME)。 -Dweblogic.StdoutDebugEnabled=false
logHome ドメイン・ログ、サーバー・ログ、サーバー出力、ノード・マネージャ・ログ、イントロスペクタ出力およびサーバーHTTPアクセス・ログ・ファイルのインポッド・ロケーション。 指定しない場合、値はdomainUIDから/shared/logs/<domainUID>として導出されます。 /shared/logs/domain1
managedServerNameBase 管理対象サーバー名の生成に使用されるベース文字列。 managed-server
managedServerPort 各管理対象サーバーのポート番号。 8001
mode createDomainWdtModelで指定されたWDTモデルを使用するか、createDomainWlstScriptで指定されたオフラインWLSTスクリプトを使用してWebLogicドメインを作成するか。 有効な値は、wdtまたはwlstです。 wdt
namespace ドメインを作成するKubernetesネームスペース。 default
persistentVolumeClaimName 永続ボリューム要求の名前。 指定しない場合、値はdomainUIDから<domainUID>-weblogic-sample-pvcとして導出されます。 domain1-weblogic-sample-pvc
productionModeEnabled 本番モードがドメインに対して有効かどうかを示すブール値。 true
serverStartPolicy 起動するWebLogic Serverインスタンスを決定します。 有効な値はNever, IfNeeded, AdminOnlyです。 IfNeeded
t3ChannelPort ネットワーク・アクセス・ポイントのT3チャネルのポート。 30012
t3PublicAddress T3チャネルのパブリック・アドレス。 これはKubernetesクラスタのパブリック・アドレスに設定する必要があります。 通常、これはロード・バランサ・アドレスです。 開発環境の場合のみ、単一サーバー(オール・イン・ワン)のKubernetesデプロイメントでは、これをマスターのアドレスに設定することも、少なくともワーカー・ノードのいずれかのアドレスに設定することもできます。 指定しない場合、スクリプトはそれをKubernetesクラスタのIPアドレスに設定しようとします。
weblogicCredentialsSecretName 管理サーバーのユーザー名およびパスワードのKubernetesシークレットの名前。 指定しない場合、値はdomainUIDから<domainUID>-weblogic-credentialsとして導出されます。 domain1-weblogic-credentials
serverPodCpuRequest, serverPodMemoryRequest, serverPodCpuCLimit, serverPodMemoryLimit 各サーバーポッドに許可されるコンピュート・リソースの最大量、および必要なコンピュート・リソースの最小量。 詳細は、Managing Compute Resources for ContainersのKubernetesのドキュメントを参照してください。 リソース・リクエストおよびリソース制限が指定されていません。
rcuCredentialsSecret データベース資格証明を含むKubernetesのシークレット。 domain1-rcu-credentials
rcuDatabaseURL データベースURL。 database:1521/service
rcuSchemaPrefix データベースで使用するスキーマ・プレフィクス(SOA1など)。 これをdomainUIDと同じにして、対応するドメインをRCUスキーマに簡略化できます。 domain1
toolsDir WebLogic Deploy ToolingおよびWebLogic Image Toolがインストールされているディレクトリ。 これらのツールがまだインストールされていない場合、スクリプトはこのディレクトリにインストールされます。 /tmp/dhii-sample/tools
wdtVersion スクリプトによってインストールされるWebLogic Deploy Toolingのバージョン。 これは、1.9.10やLATESTなどの特定のバージョンにできます。 LATEST
witVersion スクリプトによってインストールされるWebLogic Image Toolのバージョン。 これは、1.9.10やLATESTなどの特定のバージョンにできます。 LATEST

生成されたYAMLファイル内のKubernetesリソースの名前は、入力YAMLファイルに指定されたプロパティの一部の値を使用して形成される可能性があることに注意してください。 これらのプロパティには、adminServerName、clusterNameおよびmanagedServerNameBaseが含まれます。 これらの値にKubernetesサービス名に無効な文字が含まれている場合、それらの文字は生成されたYAMLファイルで有効な値に変換されます。 たとえば、大文字は小文字に変換され、アンダースコア("_")はハイフン("-")に変換されます。

このサンプルは、1つのクラスタのみを持つドメインのFMW Infrastructureドメイン・ホームおよび関連するKubernetesリソースを作成する方法を示しています。 また、このサンプルでは、ユーザーが独自のスクリプトを指定して他のユースケースのドメイン・ホームを作成する機能も提供しています。 生成されたドメイン・リソースYAMLファイルを、より多くのユース・ケースに対応するように変更することもできます。
