Domain home on a PV

サンプル・スクリプトは、既存のKubernetes PersistentVolume (PV)およびPersistentVolumeClaim (PVC)でのWebLogicドメイン・ホームの作成を示しています。 スクリプトはドメイン・リソースYAMLファイルも生成し、対応するドメインのKubernetesアーティファクトを起動するために使用できます。 オプションで、スクリプトによってドメインおよびWebLogic Serverのポッドとサービスが起動されます。

前提条件

開始する前に、このドキュメント「ドメイン・リソース」をお読みください。

ドメイン作成スクリプトを実行する前に、次の前提条件を満たす必要があります:

  • WebLogic Kubernetes Operatorが実行されていることを確認します。

  • オペレータには、パッチ29135930が適用されたOracle WebLogic Server 12.2.1.3.0、またはOracle WebLogic Server 12.2.1.4.0またはOracle WebLogic Server 14.1.1.0.0を含むWebLogicイメージが必要です。 既存のWebLogic Server一般提供イメージcontainer-registry.oracle.com/middleware/weblogic:12.2.1.3には、このパッチが適用されています。

    ノート: OracleはWebLogic Server 12.2.1.3をPSUおよびセキュリティ・パッチを提供する拡張サポートを2023年6月までの限定で提供しています。 WebLogic Server 12.2.1.3のCPUイメージは、2023年1月と2023年4月のCPUサイクルで公開されます。

    このサンプルでは、一般提供 (GA)イメージを使用しています。 GAイメージは、パブリック・インターネットから環境を使用できないデモンストレーションおよび開発目的「のみ」に適しています。「本番での使用はできません」。 本番では、常にOCRのCPU (パッチ適用済)イメージを使用するか、WebLogic Image Tool (WIT)と--recommendedPatchesオプションを使用してイメージを作成する必要があります。 詳細は、「Oracle WebLogic Serverの本番環境の保護」の「最新のパッチと更新の適用」を参照してください。

  • デフォルトのネームスペースを使用する場合を除き、ドメインのKubernetesネームスペースを作成します。

  • 同じKubernetesネームスペースで、ドメイン・ホームをホストするKubernetes PersistentVolume (PV)およびドメインのKubernetes PersistentVolumeClaim (PVC)を作成します。 PVおよびPVCを作成するサンプルについては、「サンプルPVおよびPVCの作成」を参照してください。 デフォルトでは、create-domain.shスクリプトはdomainUIDがdomain1に設定されたドメインを作成し、PVC domain1-weblogic-sample-pvcが存在することを想定しています。 create-pv-pvc.shを使用して、domainUIDがdomain1に設定された入力ファイルでdomain1-weblogic-sample-pvcを作成できます。

  • ドメインと同じKubernetesネームスペースに管理アカウントのKubernetesシークレットusernameおよびpasswordを作成します。

永続ストレージの使用に関する次の重要な考慮事項に注意してください。

永続ボリュームの作成に使用できる様々なKubernetesストレージ・プロバイダがあります。 使用しているKubernetesのバリアントおよびバージョンに応じて、永続ボリュームの作成に必要なステップは異なります。 ReadWriteManyオプションをサポートするストレージ・プロバイダを使用する必要があります。

多くのストレージ・プロバイダは、rootユーザーが所有する永続ボリュームにファイル・システムを作成します。 そのような場合は、標準のWebLogic Serverイメージで使用されるoracleユーザー(uid 1000)が永続ボリュームのファイル・システムに書き込めるように、ファイル権限または所有権を更新する必要があります。

このサンプルでは、永続ボリューム上のすべてのファイルの所有者が自動的にuid 1000に設定されます。 この動作を変更する場合は、kubernetes/samples/scripts/create-weblogic-domain/domain-home-on-pv/create-domain-job-template.yamlを編集し、initContainerセクションを編集または削除してください。

Kubernetesの一部のバリアント(OpenShiftなど)では、WebLogicプロセスが永続ボリューム上のファイル・システムにアクセスできるように、ユーザー1000で実行するようにポッドを構成することも必要になる場合があります。

スクリプトを使用してドメインを作成

create-domain-inputs.yamlファイルのコピーを作成し、入力ファイルおよび出力ディレクトリを指定して作成スクリプトを実行します:

$ ./create-domain.sh \
  -i create-domain-inputs.yaml \
  -o /<path to output-directory>

create-domain.shスクリプトとその入力ファイルは、デモンストレーション目的「のみ」用です。その内容および生成するドメイン・リソース・ファイルは、予告なしに変更される場合があります。 本番環境では、WebLogic Image ToolおよびWebLogic Deploy Tooling (適用される場合)、およびドメイン・リソース・ファイルを直接使用することを強くお勧めします。

このスクリプトは、次のステップを実行します:

  • このドメイン用に生成されたKubernetes YAMLファイルのディレクトリを作成します(まだ存在しない場合)。 パス名は/<path to output-directory>/weblogic-domains/<domainUID>です。 ディレクトリがすでに存在する場合は、このスクリプトを使用する前にその内容を削除する必要があります。
  • ユーティリティWebLogic Serverコンテナを起動し、オフラインのWLSTスクリプトまたはWebLogic Deploy Tooling (WDT)スクリプトを実行して共有ストレージにドメインを作成するKubernetesジョブを作成します。
  • ジョブが終了するまで待機します。
  • 以前に作成したディレクトリに、Kubernetesドメイン・リソースのYAMLファイルdomain.yamlを作成します。 このYAMLファイルは、kubectl create -fまたはkubectl apply -fコマンドを使用してKubernetesリソースを作成するために使用できます: 
$ kubectl apply -f /<path to output-directory>/weblogic-domains/<domainUID>/domain.yaml
  • 作成スクリプトによって作成されたドメイン・ホームをクリーンアップする便利なユーティリティ・スクリプトdelete-domain-job.yamlを作成します。

便宜上、スクリプトでは、-eオプションを使用して、オプションでドメイン・オブジェクトを作成できます。これにより、対応するWebLogic Serverポッドおよびサービスも作成されます。

作成スクリプトの使用方法は次のとおりです:

$ sh create-domain.sh -h

usage: create-domain.sh -o dir -i file [-e] [-v] [-h]
  -i Parameter inputs file, must be specified.
  -o Output directory for the generated YAML files, must be specified.
  -e Also create the resources in the generated YAML files, optional.
  -v Validate the existence of persistentVolumeClaim, optional.
  -h Help

サンプル・スクリプトを別のロケーションにコピーする場合は、<weblogic-kubernetes-operator-project>/kubernetes/samples/scriptsディレクトリ内のすべてをターゲット・ディレクトリにコピーし、元のディレクトリ階層を維持してください。

スクリプトによって作成されるデフォルト・ドメインには、次の特性があります:

  • ポート7001でリスニングするadmin-serverという名前の管理サーバー。
  • サイズ5のcluster-1という名前の動的クラスタ。
  • ポート8001でリスニングする、managed-server1およびmanaged-server2という名前の2つの管理対象サーバー。
  • /shared/logs/<domainUID>にあるログ・ファイル。
  • アプリケーションはデプロイされていません
  • データ・ソースまたはJMSリソースがありません。
  • T3チャネル。

ドメイン作成の入力は、create-domain-inputs.yamlを編集してカスタマイズできます。

serverPodCpuRequest, serverPodMemoryRequest, serverPodCpuCLimit, serverPodMemoryLimit 各サーバーポッドに許可されるコンピュート・リソースの最大量、および必要なコンピュート・リソースの最小量。 詳細は、Managing Compute Resources for ContainersのKubernetesのドキュメントを参照してください。 リソース・リクエストおよびリソース制限が指定されていません。

生成されたYAMLファイル内のKubernetesリソースの名前は、create-inputs.yamlファイルで指定されたプロパティの一部の値で構成される場合があります。 これらのプロパティには、adminServerName、clusterNameおよびmanagedServerNameBaseが含まれます。 これらの値にKubernetesサービス名に無効な文字が含まれている場合、それらの文字は生成されたYAMLファイルで有効な値に変換されます。 たとえば、大文字は小文字に変換され、アンダースコア("_")はハイフン("-")に変換されます。

このサンプルは、単一のクラスタのみを持つドメインのWebLogicドメイン・ホームおよび関連するKubernetesリソースを作成する方法を示しています。 また、このサンプルでは、ユーザーが独自のスクリプトを指定して他のユースケースのドメイン・ホームを作成する機能も提供しています。 生成されたドメイン・リソースYAMLファイルを、より多くのユース・ケースに対応するように変更することもできます。

トラブルシューティング

メッセージ: status on iteration 20 of 20 pod domain1-create-weblogic-sample-domain-job-4qwt2 status is Pending The create domain job is not showing status completed after waiting 300 seconds.
最も可能性の高い原因は、domain-home-on-pv/create-domain-inputs.yamlで定義されているpersistentVolumeClaimNameの値に関連しています。
これが問題であるかどうかを確認するには: 

* Execute `kubectl get all --all-namespaces` to find the name of the `create-weblogic-sample-domain-job`.
* Execute  `kubectl describe pod <name-of-create-weblogic-sample-domain-job>` to see if there is an event that has text similar to `persistentvolumeclaim "domain1-weblogic-sample-pvc" not found`.
* Find the name of the PVC that was created by executing [create-pv-pvc.sh](https://github.com/oracle/weblogic-kubernetes-operator/blob/release/4.0/kubernetes/samples/scripts/create-weblogic-domain-pv-pvc/create-pv-pvc.sh), using `kubectl describe pvc`. It is likely to be `weblogic-sample-pvc`.
* Change the value of `persistentVolumeClaimName` to match the name created when you executed [create-pv-pvc.sh](https://github.com/oracle/weblogic-kubernetes-operator/blob/release/4.0/kubernetes/samples/scripts/create-weblogic-domain-pv-pvc/create-pv-pvc.sh).
* Rerun the `create-domain.sh` script with the same arguments as you did before.
* Verify that the operator is deployed. Use the command:

$ kubectl  get all --all-namespaces

次のような行を探します:

weblogic-operator1   pod/weblogic-operator-

出力に類似するものが見つからない場合は、WebLogic Kubernetes Operatorが完全にインストールされていない可能性があります。 オペレータ「インストール手順」を確認します。

メッセージ: ERROR: Unable to create folder /shared/domains
最も一般的な原因は、実行時に使用した入力ファイルでのweblogicDomainStoragePathの値の選択が不適切であることです:

$ create-pv-pvc.sh

再実行する前に、「サンプル・ドメインのリソースの削除」でそのファイルの値を修正し、コマンドを再実行してPV/PVCおよび資格証明を作成する必要があります:

$ create-domain.sh

weblogicDomainStoragePathの正しい値は、次の要件を満たします:

  • ディレクトリの名前である必要があります。
  • ディレクトリはworld writableである必要があります。

オプションで、最初にサンプルを実行した後に、次のステップに従って指定したディレクトリに対する権限を制限します:

  • rootユーザーになります。
  • ls -nd $value-of-weblogicDomainStoragePath
    • 出力の3番目と4番目のフィールドの値を書き留めます。
  • chown $third-field:$fourth-field $value-of-weblogicDomainStoragePath
  • chmod 755 $value-of-weblogicDomainStoragePath
  • 通常のユーザーIDに戻ります。

メッセージ: ERROR: The create domain job will not overwrite an existing domain. The domain folder /shared/domains/domain1 already exists
このメッセージは、create-pv-pvc-inputs.yamlのweblogicDomainStoragePathの値として指定されたディレクトリにディレクトリdomains/domain1が存在する場合に表示されます。 たとえば、weblogicDomainStoragePathの値が/tmp/wls-op-4-k8sの場合、/tmp/wls-op-4-k8s/domains/domain1を削除(または移動)する必要があります。
