Domain home in image

サンプル・スクリプトでは、GitHubプロジェクト上のOracle WebLogic ServerイメージのDomain home in imageサンプルの一つを使用して、イメージ内にWebLogicドメイン・ホームを作成する方法を示しています。 サンプル・スクリプトには、WebLogicドメイン・ログ、サーバー・ログ、サーバー出力ファイルおよびノード・マネージャ・ログを既存のKubernetes PersistentVolume (PV)およびPersistentVolumeClaim (PVC)に配置するオプションがあります。 スクリプトでは、ドメインYAMLファイルも生成されます。このファイルは、スクリプトで使用したり、WebLogic Serverポッドやサービスなど、対応するドメインのKubernetesアーティファクトを起動するために手動で使用できます。

前提条件

開始する前に、このドキュメント「ドメイン・リソース」をお読みください。

ドメイン作成スクリプトを実行する前に、次の前提条件を満たす必要があります:

  • WebLogic Deploy Tooling (WDT)サンプルでは、JAVA_HOMEがJava JDKバージョン1.8以上に設定されている必要があります。
  • オペレータには、パッチ29135930が適用されたOracle WebLogic Server 12.2.1.3.0、Oracle WebLogic Server 12.2.1.4.0またはOracle WebLogic Server 14.1.1.0.0が必要です。 既存のWebLogic Serverイメージであるcontainer-registry.oracle.com/middleware/weblogic:12.2.1.3には、必要なパッチがすべて適用されています。 イメージを取得または作成する方法の詳細は、「WebLogic Serverイメージ」を参照してください。
  • デフォルトのネームスペースを使用する場合を除き、ドメインのKubernetesネームスペースを作成します。
  • logHomeOnPVが有効になっている場合は、ログ・ホームをホストするKubernetes PersistentVolumeと、同じKubernetesネームスペース内のドメインのKubernetes PersistentVolumeClaimを作成します。 PVおよびPVCを作成するサンプルについては、「サンプルPVおよびPVCの作成」を参照してください。
  • フィールドusernameおよびpasswordを含むWebLogic管理者資格証明のKubernetesシークレットを作成し、シークレット名がweblogicCredentialsSecretNameに指定された値と一致することを確認します。次の「構成パラメータ」を参照してください。 例えば: 
$ cd ./kubernetes/samples/scripts/create-weblogic-domain-credentials
$ create-weblogic-credentials.sh
   -u <username>
   -p <password>
   -d domain1
   -n default
   -s domain1-weblogic-credentials

ノート: この例を使用して、weblogicCredentialsSecretNameをdomain1-weblogic-credentialsとして構成します。

スクリプトを使用してドメインを作成

create-domain.shスクリプトは、新しいドメイン・ホームとその中の異なる内部domain secretを使用して、実行ごとに新しいコンテナ・イメージを生成します。 同じドメイン内に異なるドメイン・シークレットを持つ異なるイメージが存在しないようにするには、新しいドメインでアクティブなドメインとは異なるdomainUIDを使用するか、次のコマンドを使用して既存のドメインを削除し、すべてのWebLogic Serverインスタンスポッドが終了するまで待機してから、同じdomainUIDを持つドメインを作成することを強くお薦めします: $ kubectl delete domain [domainUID] -n [domainNamespace]

ドメインを作成するサンプルは、次のディレクトリにあります:

$ cd kubernetes/samples/scripts/create-weblogic-domain/domain-home-in-image

create-domain-inputs.yamlファイルのコピーを作成し、正しい値で更新し、入力ファイルおよび出力ディレクトリを指定して作成スクリプトを実行します:

$ ./create-domain.sh \
  -u <username> \
  -p <password> \
  -i create-domain-inputs.yaml \
  -o /<path to output-directory>

このスクリプトは、次のステップを実行します:

  • このドメイン用に生成されたプロパティおよびKubernetes YAMLファイルのディレクトリを作成します(まだ存在しない場合)。 パス名は/<path to output-directory>/weblogic-domains/<domainUID>です。 ディレクトリがすでに存在する場合は、その内容が削除されます。
  • 前述の手順で作成したディレクトリにプロパティ・ファイルdomain.propertiesを作成します。 このプロパティ・ファイルは、サンプルWebLogic Serverドメインを作成するために使用されます。
  • git clone https://github.com/oracle/docker-images.gitを使用して、domainHomeImageBuildPathプロパティから導出されたディレクトリにWebLogic docker-imagesプロジェクトをクローンします。 デフォルトでは、スクリプトは常にディレクトリをクリーンアップし、実行のたびに再度クローンします。 以前にクローンしたプロジェクトを使用する場合は、-kオプションを指定する必要があります。 指定されたdomainHomeImageBuildPathが空の場合、-kオプションが指定されていても、スクリプトはプロジェクトをクローンします。
  • properties/docker-build/domain_security.propertiesファイルの組込みユーザー名およびパスワードを、-uおよび-pオプションを使用してコマンドラインで指定されたusernameおよびpasswordに置き換えます。 これらの資格証明は、create-domain-inputs.yamlファイルのweblogicCredentialsSecretNameプロパティを使用して指定されたシークレットのWebLogicドメイン管理者資格証明と一致する必要があります。
  • Dockerサンプル、「Oracle WebLogic Scripting Tooling (WLST)を使用したWebLogic Serverドメインを含むサンプル・イメージ」または「Oracle WebLogic Deploy Tooling (WDT)を使用したWebLogic Serverドメインを含むイメージの例」に基づいてコンテナ・イメージを構築します。 コンテナ・イメージにサンプルのWebLogic Serverドメインが作成されます。

    Oracleは、ドメイン・ホームを含むイメージをプライベートとしてレジストリ(Oracle Cloud Infrastructure Registry、GitHub Container Registryなど)に格納することを強くお薦めします。これは、このイメージには、外部リソースへのアクセスに使用されるキーや資格証明(データ・ソース・パスワードなど)など、ドメインに関する機密情報が含まれるためです。 詳細は、「WebLogic domain in image保護」を参照してください。

  • 生成されるイメージを参照するタグを作成します。
  • 前述の手順で作成したディレクトリに、KubernetesドメインYAMLファイルdomain.yamlを作成します。 このYAMLファイルは、kubectl create -fまたはkubectl apply -fコマンドを使用してKubernetesリソースを作成するために使用できます。 
$ kubectl apply -f /<path to output-directory>/weblogic-domains/<domainUID>/domain.yaml

便宜上、スクリプトでは、-eオプションを使用して、オプションでドメイン・オブジェクトを作成できます。これにより、対応するWebLogic Serverポッドおよびサービスが作成されます。 このオプションは、単一ノードのKubernetesクラスタでのみ使用する必要があります。

複数ノードのKubernetesクラスタの場合は、kubectl apply -fコマンドを使用してドメインYAMLファイルを作成する前に、生成されたイメージがすべてのノードで使用可能であることを確認します。

作成スクリプトの使用方法は次のとおりです:

$ sh create-domain.sh -h
usage: create-domain.sh -o dir -i file -u username -p password [-k] [-e] [-h]
  -i Parameter inputs file, must be specified.
  -o Ouput directory for the generated properties and YAML files, must be specified.
  -u User name used in building the image for WebLogic domain in image.
  -p Password used in building the image for WebLogic domain in image.
  -e Also create the resources in the generated YAML files, optional.
  -v Validate the existence of persistentVolumeClaim, optional.
  -k Keep what has been previously cloned from https://github.com/oracle/docker-images.git, optional.
     If not specified, this script will always remove the existing project and clone again.
  -h Help

サンプル・スクリプトを別のロケーションにコピーする場合は、<weblogic-kubernetes-operator-project>/kubernetes/samples/scriptsディレクトリ内のすべてをターゲット・ディレクトリにコピーし、元のディレクトリ階層を維持してください。

スクリプトによって作成されるデフォルト・ドメインには、次の特性があります:

  • ポート7001でリスニングするadmin-serverという名前の管理サーバー。
  • サイズ5のcluster-1という名前の動的クラスタ。
  • ポート8001でリスニングする、managed-server1およびmanaged-server2という名前の2つの管理対象サーバー。
  • アプリケーションはデプロイされていません
  • T3チャネル。

Kubernetesクラスタに対してリモートのマシンからサンプルを実行し、クラスタに対してローカルなレジストリに新しいイメージをプッシュする必要がある場合は、次の手順を実行する必要があります(「構成パラメータ」表のimageプロパティも参照)。):

  • 入力ファイルのimageプロパティをターゲット・イメージ名(必要に応じて、レジストリ・ホスト名、ポートおよびタグを含む)に設定します。
  • Kubernetesでプライベート・レジストリからイメージをプルする場合は、資格証明を保持するKubernetesシークレットを作成し、入力ファイルのimagePullSecretNameプロパティをシークレットの名前に設定します。

    Kubernetesのシークレットは、ドメインが実行されるネームスペースと同じネームスペースに存在する必要があります。 詳細は、「WebLogic domain in image保護」を参照してください。

  • -eオプションを指定せずにcreate-domain.shスクリプトを実行します。
  • imageをターゲット・レジストリにプッシュします。
  • 次のコマンドを実行してドメインを作成します:
$ kubectl apply -f /<path to output-directory>/weblogic-domains/<domainUID>/domain.yaml

ドメイン作成の入力は、create-domain-inputs.yamlを編集してカスタマイズできます。

構成パラメータ

次のパラメータを入力ファイルに指定できます。

パラメータ 定義 デフォルト
sslEnabled 各WebLogic Serverインスタンスに対してSSLを有効にするかどうかを示すブール値。 false
adminPort Kubernetesクラスタ内の管理サーバーのポート番号。 7001
adminServerSSLPort Kubernetesクラスタ内の管理サーバーのSSLポート番号。 7002
adminNodePort Kubernetesクラスタ外の管理サーバーのポート番号。 30701
adminServerName 管理サーバーの名前。 admin-server
clusterName ドメインに対して生成するWebLogicクラスタ・インスタンスの名前。 cluster-1
configuredManagedServerCount ドメインに対して生成する管理対象サーバー・インスタンスの数。 5
domainHomeImageBase WebLogicドメイン・イメージの構築に使用されるベースWebLogicバイナリ・イメージ。 オペレータには、パッチ29135930が適用されたOracle WebLogic Server 12.2.1.3.0、Oracle WebLogic Server 12.2.1.4.0またはOracle WebLogic Server 14.1.1.0.0が必要です。 既存のWebLogic Serverイメージであるcontainer-registry.oracle.com/middleware/weblogic:12.2.1.3には、必要なパッチがすべて適用されています。 イメージを取得または作成する方法の詳細は、「WebLogic Serverイメージ」を参照してください。 container-registry.oracle.com/middleware/weblogic:12.2.1.3
domainHomeImageBuildPath https://github.com/oracle/docker-images.gitプロジェクト内のWebLogicのdomain home in imageイメージのロケーション。 指定しない場合は、./docker-images/OracleWebLogic/samples/12213-domain-home-in-imageを使用します。 別の可能な値は、WLSTのかわりにWDTを使用してドメイン構成を生成する./docker-images/OracleWebLogic/samples/12213-domain-home-in-image-wdtです。 ./docker-images/OracleWebLogic/samples/12213-domain-home-in-image
domainPVMountPath ドメイン永続ボリュームのマウント・パス。 logHomeOnPVがtrueの場合、このパラメータは必須です。 それ以外の場合、このパラメータは無視されます。 /shared
domainUID この特定のドメインの識別に使用される一意のID。 生成されたWebLogicドメインの名前およびドメインの名前として使用されます。 このIDは、Kubernetesクラスタ内のすべてのドメインで一意である必要があります。 このIDには、Kubernetesサービス名に無効な文字を含めることはできません。 domain1
exposeAdminNodePort 管理サーバーがKubernetesクラスタの外部に公開されているかどうかを示すブール値。 false
exposeAdminT3Channel T3管理チャネルがKubernetesクラスタの外部に公開されているかどうかを示すブール値。 false
httpAccessLogInLogHome logHomeOnPVがtrueの場合に、サーバーのHTTPアクセス・ログ・ファイルをlogHomeと同じディレクトリに書き込むかどうかを示すブール値。 それ以外の場合、サーバーのHTTPアクセス・ログ・ファイルは、WebLogicドメイン・ホーム構成で指定されたディレクトリに書き込まれます。 true
image オペレータがドメインの起動に使用するWebLogic Serverイメージ。 ドメイン作成スクリプトは、ドメイン・ホームを含むWebLogic Serverイメージを生成します。 デフォルトでは、スクリプトは、生成されたWebLogic ServerイメージをdomainHomeImageBuildPathプロパティに基づいてdomain-home-in-imageまたはdomain-home-in-image-wdtとしてタグ付けし、domainHomeImageBaseから取得したタグを追加して使用し、生成されたドメインYAMLファイルにimage要素を設定します。 このプロパティが設定されている場合、ドメインの作成スクリプトは、デフォルト値のかわりに指定された値を使用して、生成されたイメージにタグを付け、ドメインYAMLファイルでimageを設定します。 スクリプトを使用して作成されるドメインごとに一意の値が必要です。 ドメインを実行するKubernetesクラスタに対してリモートのマシンからサンプル・スクリプトを実行する場合は、このプロパティを、そのKubernetesクラスタに対してローカルなレジストリで使用するイメージ名に設定する必要があります。 また、kubectl create -fまたはkubectl apply -fコマンドを使用してドメインを起動する前に、imageをそのレジストリにプッシュする必要があります。
imagePullPolicy WebLogic Serverイメージ・プル・ポリシー。 有効な値は、IfNotPresent、AlwaysまたはNeverです。 IfNotPresent
imagePullSecretName WebLogic Serverイメージをプルするためにコンテナ・レジストリにアクセスするKubernetesシークレットの名前。 このパラメータを指定すると、シークレットの存在が検証されます。
includeServerOutInPodLog ポッドのstdoutにサーバー.out intを含めるかどうかを示すブール値。 true
initialManagedServerReplicas ドメインに対して最初に起動する管理対象サーバーの数。 2
javaOptions 管理サーバーおよび管理対象サーバーを起動するためのJavaオプション。 Javaオプションには、WebLogicドメイン情報を取得するために、次の事前定義済変数のいずれかを参照できます: $(DOMAIN_NAME), $(DOMAIN_HOME), $(ADMIN_NAME), $(ADMIN_PORT)、および$(SERVER_NAME)。 sslEnabledがtrueに設定され、WebLogicデモ証明書が使用されている場合は、-Dweblogic.security.SSL.ignoreHostnameVerification=trueを追加して、起動時に管理対象サーバーが管理サーバーに接続できるようにします。 この環境で生成されたWebLogicデモ証明書には、通常、ランタイム・コンテナのホスト名とは異なるホスト名が含まれています。 -Dweblogic.StdoutDebugEnabled=false
logHomeOnPV ログ・ホームを永続ボリュームに格納するかどうかを指定します。 trueに設定した場合、logHome、persistentVolumeClaimNameおよびdomainPVMountPathパラメータを指定する必要があります。 false
logHome ドメイン・ログ、サーバー・ログ、サーバー出力、ノード・マネージャ・ログ、イントロスペクタ出力およびサーバーHTTPアクセス・ログ・ファイルのインポッド・ロケーション。 指定しない場合、値はdomainUIDから/shared/logs/<domainUID>として導出されます。 logHomeOnPVがtrueの場合、このパラメータは必須です。 それ以外の場合、このパラメータは無視されます。 /shared/logs/domain1
managedServerNameBase 管理対象サーバー名の生成に使用されるベース文字列。 managed-server
managedServerPort 各管理対象サーバーのポート番号。 8001
managedServerSSLPort 各管理対象サーバーのSSLポート番号。 8002
namespace ドメインを作成するKubernetesネームスペース。 default
persistentVolumeClaimName 永続ボリューム要求の名前。 指定しない場合、値はdomainUIDから<domainUID>-weblogic-sample-pvcとして導出されます。 logHomeOnPVがtrueの場合、このパラメータは必須です。 それ以外の場合、このパラメータは無視されます。 domain1-weblogic-sample-pvc
productionModeEnabled 本番モードがドメインに対して有効かどうかを示すブール値。 true
serverStartPolicy 起動するWebLogic Serverインスタンスを決定します。 有効な値はNEVER, IF_NEEDED, ADMIN_ONLYです。 IF_NEEDED
t3ChannelPort ネットワーク・アクセス・ポイントのT3チャネルのポート。 30012
t3PublicAddress T3チャネルのパブリック・アドレス。 これはKubernetesクラスタのパブリック・アドレスに設定する必要があります。 通常、これはロード・バランサ・アドレスです。 開発環境の場合のみ、単一サーバー(オール・イン・ワン)のKubernetesデプロイメントでは、これをマスターのアドレスに設定することも、少なくともワーカー・ノードのいずれかのアドレスに設定することもできます。 指定しない場合、スクリプトはそれをKubernetesクラスタのIPアドレスに設定しようとします。
weblogicCredentialsSecretName 管理サーバーのユーザー名およびパスワードのKubernetesシークレットの名前。 domain1-weblogic-credentials
serverPodCpuRequest, serverPodMemoryRequest, serverPodCpuCLimit, serverPodMemoryLimit 各サーバーポッドに許可されるコンピュート・リソースの最大量、および必要なコンピュート・リソースの最小量。 詳細は、Managing Compute Resources for ContainersのKubernetesのドキュメントを参照してください。 リソース・リクエストおよびリソース制限が指定されていません。

生成されたYAMLファイル内のKubernetesリソースの名前は、create-inputs.yamlファイルで指定されたプロパティの一部の値で構成される場合があります。 これらのプロパティには、adminServerName、clusterNameおよびmanagedServerNameBaseが含まれます。 これらの値にKubernetesサービス名に無効な文字が含まれている場合、それらの文字は生成されたYAMLファイルで有効な値に変換されます。 たとえば、大文字は小文字に変換され、アンダースコア("_")はハイフン("-")に変換されます。

このサンプルでは、単一のクラスタのみを持つドメインのWebLogicドメイン・ホームおよび関連するKubernetesリソースを作成する方法を示します。 また、このサンプルでは、ユーザーが独自のスクリプトを指定して他のユースケースのドメイン・ホームを作成する機能も提供しています。 また、生成されたドメインYAMLファイルは、より多くのユースケースに対応するように変更できます。
