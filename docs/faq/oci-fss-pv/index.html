永続ボリュームに対するOCIファイル・ストレージ(FSS)の使用

Oracle Container Engine for Kubernetes (通常OKEと呼ばれる)でKubernetesクラスタを実行しており、永続ボリューム用のOCIファイル・ストレージ(FSS)を使用してWebLogicドメイン・ホームを格納する場合、オペレータの永続ボリューム・サンプルに示されているように、ドメインの初期化時に永続ボリュームのファイル所有権を適切に初期化するには更新が必要です。

永続ボリュームでのファイル権限の処理は、クラウド・プロバイダ間で異なる場合があり、Linuxベースのシステムでの基礎となるストレージの処理でも異なる場合があります。 次の手順では、UID 1000およびGID 1000が通常oracleまたはopcユーザーを表す場合に、標準のOracleイメージで使用されるファイル所有権を更新するオプションを提供します。 永続ボリュームの処理の詳細は、「永続ストレージ」を参照してください。

永続ボリューム・サンプルを使用したドメイン作成中に障害が発生しました

「永続ボリュームでのドメイン・ホームの作成」の既存のサンプルでは、Kubernetesジョブを使用してドメインを作成します。 このサンプルでは、initContainersセクションを使用して、OKEクラスタで使用されるOCI FSS作成ボリュームで失敗するファイル所有権を変更します。

OCI FSSボリュームには変更できないファイルが含まれているため、Kubernetesジョブは失敗します。 失敗は、Kubernetesジョブ・ポッドの説明に表示されます:

永続ボリューム・サンプルのドメインの更新

create-domain-job-template.yamlの次のスニペットでは、initコンテナの更新されたcommandを確認できます:

apiVersion: batch/v1
kind: Job
metadata:
  name: %DOMAIN_UID%-create-weblogic-sample-domain-job
  namespace: %NAMESPACE%
spec:
  template:
    metadata:
  :
  :
    spec:
      restartPolicy: Never
      initContainers:
        - name: fix-pvc-owner
          image: %WEBLOGIC_IMAGE%
          command: ["sh", "-c", "chown 1000:0 %DOMAIN_ROOT_DIR%/. && find %DOMAIN_ROOT_DIR%/. -maxdepth 1 ! -name '.snapshot' ! -name '.' -print0 | xargs -r -0 chown -R 1000:0"]
          volumeMounts:
          - name: weblogic-sample-domain-storage-volume
            mountPath: %DOMAIN_ROOT_DIR%
          securityContext:
            runAsUser: 0
            runAsGroup: 0
      containers:
        - name: create-weblogic-sample-domain-job
          image: %WEBLOGIC_IMAGE%

  :

この新しいcommandをこのテンプレート・ファイルのコピーで使用します。 これにより、WebLogicドメインが永続ボリュームに作成される前に、予想されるファイルの所有権のみが更新されます。
