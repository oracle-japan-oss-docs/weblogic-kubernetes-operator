ターゲット環境

「ドメインの検出」および「モデルの準備」 Toolsを使用すると、特定のターゲット環境と互換性を持つように作成されたモデルおよびその他のファイルをカスタマイズできます。 ターゲット環境のオプションには、次のものがあります:

  • モデル内の一部の属性に対するモデル・トークンの使用。 「モデル・トークン」を参照してください。
  • モデル内の資格証明にKubernetesシークレットを使用します。
  • モデルへのフィルタの適用。 「モデル・フィルタ」を参照してください。
  • ターゲット・システムの追加構成ファイルの作成。

ターゲット環境の指定

各ツールでは、コマンドライン引数-target <target-name>を使用してターゲット環境を指定します。ここで、<target-name>は事前構成済のターゲット環境またはユーザー定義環境を参照します。 また、-output_dir <output-directory>引数では、ターゲット環境のファイルの格納場所を指定します。

このコマンドラインは、ドメインの検出ツールでこれらの引数を使用する方法を示しています:

 $ $WLSDEPLOY_HOME/bin/discoverDomain.sh ... -target k8s -output_dir /etc/files

この例では、k8sターゲット・タイプを検出結果に適用し、それらのファイルを/etc/filesに配置します。

-variable_file引数を使用してツールのコマンドラインで変数ファイルを指定すると、注入された変数がそのファイルに追加されます。 変数ファイルが指定されていない場合、注入された変数はファイル<output-directory>/<target_name>_variable.propertiesに書き込まれます。

ターゲット構成ファイル

ターゲット環境は、次のロケーションにあるJSONファイルで構成されます:

$WLSDEPLOY_HOME/lib/target/<target-name>/target.json

<target-name>値は、ツールのコマンドラインの-target引数の値に対応します。 WLSインストールには、次の2つの事前定義済ターゲットが含まれています:

  • Weblogic Kubernetes Operator (named k8s)
  • Verrazzano (named vz)

新しいターゲット環境または拡張ターゲット環境は、前述のロケーションにある新しいtarget-nameを使用するか、$WDT_CUSTOM_CONFIG/target/<my-target-name>/target.jsonなどの「カスタム構成」ディレクトリを使用して定義できます。

ターゲット環境ファイルの例を次に示します:

{
  "model_filters" : {
        "discover": [
          { "name": "k8s_prep", "path": "@@TARGET_CONFIG_DIR@@/k8s_operator_filter.py" }
        ]
      },
  "variable_injectors" : {"PORT": {},"HOST": {},"URL": {}},
  "validation_method" : "lax",
  "credentials_method" : "secrets",
  "wls_credentials_name" : "__weblogic-credentials__",
  "additional_output" : "binding.yaml,model.yaml"
}

この例の各フィールドはオプションで、カスタマイズできます。

model_filters

このフィールドでは、結果のモデルに適用されるフィルタを指定します。 これは、「モデル・フィルタ」構成と同じ形式およびルールに従います。 ここでは、常にdiscoverタイプを使用する必要があります。

@@TARGET_CONFIG_DIR@@トークンを使用して、指定したフィルタがターゲット構成ファイルと同じディレクトリにあることを示すことができます。

variable_injectors

このフィールドでは、結果のモデルに適用される変数インジェクタを指定します。 これは、「可変インジェクタ」構成と同じ形式およびルールに従います。

validation_method

このフィールドを使用して、結果モデルの検証レベルを設定できます。 値laxのみが現在サポートされています。 lax検証では、結果のモデルで参照される変数およびKubernetesシークレットは、モデルの作成時に使用可能である必要はありません。

credentials_method

このフィールドでは、モデル内の資格証明の処理方法を指定します。 次の2つの値を使用できます:

  • secrets - モデル内の資格証明がKubernetesシークレットへの参照に置き換えられ、それらのシークレットを作成するUNIXスクリプトが生成されます。
  • config_override_secrets - モデル内の資格証明がpassword1などのプレースホルダー値に置き換えられ、対応するKubernetesシークレットを作成するUNIXスクリプトが生成されます。

どちらの場合も、Kubernetesシークレットを作成するスクリプトは<output-directory>/create_k8s_secrets.shに書き込まれます。 実行する前に、このスクリプトを資格証明の値で更新する必要があります

wls_credentials_name

このフィールドでは、ドメインの管理資格証明シークレットを識別するためにWDT_MODEL_SECRETS_NAME_DIR_PAIRS環境変数で使用する名前を指定します。 これは、これらのシークレットが<directory>/<name>/<key>の規則に従っていないディレクトリに格納されている場合に便利です。 WDT_MODEL_SECRETS_NAME_DIR_PAIRS環境変数の使用方法の詳細は、「モデル・トークン」を参照してください。

additional_output

このフィールドを使用して、ターゲット環境で使用する追加出力を作成できます。 この値は、$WLSDEPLOY_HOME/lib/target/<target-name>ディレクトリ内のテンプレート・ファイルのカンマ区切りリストです。 これらのテンプレートには、モデルから導出された情報が移入され、指定された出力ディレクトリに同じ名前のファイルに書き込まれます。

テンプレート・ファイルは、特定の環境に合せてカスタマイズできます。 推奨されるメソッドは、前述のように、元のテンプレートをカスタム構成ディレクトリ($WDT_CUSTOM_CONFIG/target/<target-name>/model.yamlなど)にコピーすることです。 コピーしたファイルは、参照用に元のファイルを保持しながら、必要に応じて編集できます。

事前構成済のターゲット環境

これらのターゲット環境構成は、WebLogic Deploy Toolingインストールに含まれています。

WebLogic Kubernetes Operatorターゲット

このターゲット環境は、コマンドライン引数-target wkoを指定することで適用できます。 次の追加処理が提供されます:

  • wko_operator_filter.pyフィルタは、Kubernetes環境と互換性のないモデル要素を削除するために適用されます
  • 変数は、ポート、ホストおよびURL属性のモデルに注入されます
  • 結果のモデルにlax検証が適用されます
  • モデル内の資格証明はKubernetesシークレットへの参照に置き換えられ、それらのシークレットを作成するスクリプトが生成されます
  • モデルから導出されたクラスタおよびネーミング情報とともに、追加のKubernetesリソース・ファイルmodel.yamlが生成されます

ベラッツァーノのターゲット

このターゲット環境は、コマンドライン引数-target vzを指定することで適用できます。 次の追加処理が提供されます:

  • vz_filter.pyフィルタは、Kubernetes環境と互換性のないモデル要素を削除するために適用されます
  • 変数は、ポート、ホストおよびURL属性のモデルに注入されます
  • 結果のモデルにlax検証が適用されます
  • モデル内の資格証明はプレースホルダー値に置き換えられ、対応するシークレットを作成するスクリプトが生成されます
  • モデルから導出されたクラスタおよびデータ・ソース情報を使用して、model.yamlおよびbinding.yamlという追加のKubernetesリソース・ファイルが生成されます

汎用Kubernetesターゲット

このターゲット環境は、コマンドライン引数-target k8sを指定することで適用できます。 次の追加処理が提供されます:

  • k8s_operator_filter.pyフィルタは、Kubernetes環境と互換性のないモデル要素を削除するために適用されます
  • 変数は、ポート、ホストおよびURL属性のモデルに注入されます
  • 結果のモデルにlax検証が適用されます
  • モデル内の資格証明はKubernetesシークレットへの参照に置き換えられ、それらのシークレットを作成するスクリプトが生成されます
