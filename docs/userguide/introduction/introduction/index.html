機械翻訳について

開始

オペレータは、Kubernetesを拡張して複雑なアプリケーションのインスタンスを作成、構成および管理するアプリケーション固有のコントローラです。 Oracle WebLogic Server Kubernetes Operatorは、標準のKubernetesオペレータ・パターンに従い、WebLogicドメインおよびデプロイメントの管理および操作を簡略化します。

Kubernetesクラスタには、それぞれ1つ以上のWebLogicドメインを管理する1つ以上のオペレータを含めることができます。 オペレータのインストールと構成を管理するHelmチャートが用意されています。 詳細な手順は、「こちら」を参照してください。

オペレータの前提条件

現在の本番リリースの3.1.3の場合:

  • Kubernetes 1.14.8+、1.15.7+、1.16.0+、1.17.0+、および1.18.0+(kubectl versionで確認)。
  • OpenShift 4.3システムでのFlannelネットワークv0.9.1-amd64以降 (docker images | grep flannelで確認)、Calicoネットワーク(Calico v3.16.1)またはOpenShift 4.3システムのOpenShift SDN。
  • Docker 18.9.1または19.03.1 (docker versionで確認)またはCRI-O 1.14.7 (crictl version | grep RuntimeVersionで確認)。
  • Helm 3.1.3+(helm version --client --shortで確認してください)。
  • パッチ29135930を適用したOracle WebLogic Server 12.2.1.3.0、Oracle WebLogic Server 12.2.1.4.0またはOracle WebLogic Server 14.1.1.0.0のいずれか。
    • 既存のWebLogic Serverイメージであるcontainer-registry.oracle.com/middleware/weblogic:12.2.1.3 には、必要なパッチがすべて適用されています。
    • docker run container-registry.oracle.com/middleware/weblogic:12.2.1.3 sh -c 'source $ORACLE_HOME/wlserver/server/bin/setWLSEnv.sh > /dev/null 2>&1 && java weblogic.version'を使用してWLSのバージョンを確認します。
    • docker run container-registry.oracle.com/middleware/weblogic:12.2.1.3 sh -c '$ORACLE_HOME/OPatch/opatch lspatches'を使用してWLSパッチを確認します。
  • オペレータをインストールするには、cluster-adminのロールが必要です。 オペレータは、実行時にcluster-adminロールを必要としません。
  • 現在、Linux以外のコンテナでのWebLogicの実行はサポートされていません。

クラウド・プロバイダ

Oracle 「グローバル価格設定およびライセンス・サイト」には、ライセンス・プラクティスおよびポリシーの詳細が記載されています。 WebLogic Serverおよびオペレータは、「このOracleライセンス・ポリシー」および「この適格製品リスト」で定義されている「Authorized Cloud Environments」でサポートされています。

サポートされている構成を定義する公式ドキュメントは、「こちら」です。

これらのポリシーに従って、オペレータおよびWebLogic Serverは、Oracle Container Engine for Kubernetesを使用してOracle Cloud Infrastructureでサポートされるか、OCI Compute上で「Kubernetesで使用するOracle Linux Container Services」を実行しているクラスタおよび「Authorized Cloud Environments」でサポートされます。

Microsoft Azure Kubernetes Service

Azure Kubernetes Service (AKS)は、ホスティングされたKubernetes環境です。 WebLogic Server Kubernetes Operator、Oracle WebLogic Sever 12cおよびOracle Fusion Middleware Infrastructure 12cは、Azure Kubernetes Serviceで完全にサポートおよび動作保証されています(前述のドキュメントを参照)。

AKSのサポートと制限:

  • Domain in ImageとDomain in PVの両方のドメイン・ホーム・ソース・タイプがサポートされています。
  • Domain in PVでは、永続的なボリューム要求を介してアクセスされるAzure Filesボリュームをサポートしています。「こちら」を参照してください。
  • Azureロード・バランサは、type=LoadBalancerのKubernetesサービスを使用してプロビジョニングする場合にサポートされます。
  • Oracle Cloud Infrastructureで実行されているOracleデータベースは、OCI FastConnectを介してアクセスする場合にのみ、Fusion Middleware Infrastructure MDSデータ・ストアでサポートされます。
  • Windows Serverコンテナは現在サポートされておらず、Linuxコンテナのみがサポートされています。

VMware Tanzu Kubernetes Service

Tanzu Kubernetes Grid (TKG)は、Kubernetesクラスタを迅速にデプロイおよび管理できる管理対象Kubernetesサービスです。 WebLogic Server Kubernetes OperatorおよびOracle WebLogic Severは、VMware Tanzu Kubernetes Grid Multicloud 1.1.3 ( vSphere 6.7 U3を含む)で完全にサポートおよび認定されています。

TKGのサポートと制限:

  • Domain in ImageとModel in Imageの両方のドメイン・ホーム・ソース・タイプがサポートされています。 Domain in PVはサポートされていません。
  • VSphere CSIドライバは、Read-Write-Onceポリシーを持つボリュームのみをサポートします。 PVでのストアの書込みは許可されません。
    • HAを必要とするアプリケーションの場合は、データベース内のJMSおよびJTAストアを使用します。
  • MetalLBロード・バランサでは、証明に使用されるイングレスはNGINXです。

Oracle Linux Cloud Native Environment (OLCNE)

Oracle Linux Cloud Native Environmentは、クラウド・ネイティブ・アプリケーションの開発および管理のための完全に統合されたスイートです。 Oracle Linux Cloud Native Environmentは、Open Container Initiative (OCI)およびCloud Native Computing Foundation (CNCF)標準に基づいて、マイクロサービスを編成するための主要な機能のインストール、更新、アップグレードおよび構成のための簡略化されたフレームワークを提供します。

WebLogic ServerおよびWebLogic Server Kubernetes Operatorは、Oracle Linux Cloud Native Environmentで動作保証およびサポートされています。 Operator 2.6.0は、Kubernetes 1.17.0でOLCNE 1.1の動作保証されたサポートを提供します。

OpenShift

Operator 2.0.1+は、OpenShift Container Platform 3.11.43+およびKubernetes 1.11.5 +での使用が動作保証されています。

Operator 2.5.0+は、OpenShift Container Platform 4.3.0+およびKubernetes 1.16.2 +での使用が動作保証されています。

OpenShiftでオペレータを使用する場合、WebLogicコンテナがドメイン・ファイル・システムに対する適切な権限を持つUNIX UIDで実行されるようにするには、セキュリティ・コンテキスト制約が必要です。 これは、anyuid SCC、またはユーザー/グループ1000に対して定義したカスタムSCCのいずれかです。 詳細は、セキュリティの項のOpenShiftを参照してください。

開発に重点を置いたKubernetesディストリビューションに関する重要なノート

Kubernetesには、kind、Minikube、Minishiftなど、開発に重点を置いた多数のディストリビューションがあります。 多くの場合、これらは開発マシン上の仮想マシンでKubernetesを実行します。 これらのディストリビューションでは、次のような領域に追加の課題があることがわかりました:

  • 個別のコンテナ・イメージ・キャッシュにより、イメージを保存/ロードしてDockerファイル・システム間で移動する必要があります
  • WebLogicを実行したり、必要なイメージを保持したりするには小さすぎるデフォルトの仮想マシン・ファイル・サイズおよびリソース制限
  • オペレータまたはWebLogicが依存する機能を常にサポートしているわけではないストレージ・プロバイダ
  • オペレータまたはWebLogicが依存する機能を必ずしもサポートしていないロード・バランシング実装

そのため、これらのディストリビューションを使用してオペレータまたはWebLogicを実行することをお薦め「しません」。また、WebLogicまたはこれらのディストリビューションで実行されるオペレータはサポートされません。

Docker for Desktopには同じ制限がないようですが、開発/テスト・オプションとしてサポートされています。
