オペレータのインストール

オペレータは、Helmを使用して必要なリソースを作成し、そのオペレータをKubernetesクラスタにデプロイします。 このドキュメントでは、オペレータをインストール、アップグレード、およびアンインストールする方法について説明します。

  • オペレータHelmチャートのインストール

オペレータHelmチャートのインストール

helm installコマンドを使用して、オペレータHelmチャートをインストールします。 この一部として、オペレータのリリース名を指定する必要があります。

次のいずれかを実行して、チャートのデフォルトの構成値をオーバーライドできます:

  • オーバーライドする値を含むカスタムYAMLファイルを作成し、Helmコマンドラインで--valueオプションを指定します。
  • --setオプションを使用して、個々の値をHelmコマンドラインで直接オーバーライドします。

helm installコマンドラインから--namespace引数を指定して、オペレータがインストールされるネームスペースを指定します。 指定しない場合、デフォルトでdefaultに設定されます。 ネームスペースがまだ存在しない場合は、Helmによって自動的に作成されますが(および新しいネームスペースにデフォルトのサービス・アカウントが作成されます)、リリースのアンインストール時に削除されません。 ネームスペースがすでに存在する場合は、Helmによって使用されます。 これらは、Helmの標準的な動作です。

同様に、デフォルトのserviceAccount構成値をオーバーライドして、オペレータのネームスペースでサービス・アカウントを指定することもできます(この場合、オペレータで使用されます)。 指定しない場合、デフォルトでdefaultに設定されます(たとえば、ネームスペースのデフォルトのサービス・アカウント)。 別のサービス・アカウントを使用する場合は、オペレータのHelmチャートをインストールする前に、オペレータのネームスペースとサービス・アカウントを作成する必要があります。

たとえば、Helm 3.xを使用すると、次のようになります:

$ kubectl create namespace weblogic-operator-namespace

$ helm install weblogic-operator kubernetes/charts/weblogic-operator \
  --namespace weblogic-operator-namespace \
  --values custom-values.yaml --wait

または

$ helm install weblogic-operator kubernetes/charts/weblogic-operator \
  --namespace weblogic-operator-namespace \
  --set "javaLoggingLevel=FINE" --wait

これにより、weblogic-operator-namespaceネームスペースにweblogic-operatorという名前のHelmリリースが作成され、オペレータのデプロイメントおよびサポート・リソースが構成されます。

オペレータのインストールを検証するには、helm installコマンドの出力を調べます。

オペレータ・イメージがプライベート・レジストリに格納されている場合のレジストリ資格証明の指定の詳細は、「オペレータ・イメージ・プル・シークレット」を参照してください。

または、GitHubチャート・リポジトリからオペレータHelmチャートをインストール

このリポジトリをHelmインストールに追加します:

$ helm repo add weblogic-operator https://oracle.github.io/weblogic-kubernetes-operator/charts

リポジトリが正しく追加されたことを確認します:

$ helm repo list
NAME           URL
weblogic-operator    https://oracle.github.io/weblogic-kubernetes-operator/charts

チャート・リポジトリからのチャートに関する最新情報を把握します:

$ helm repo update

リポジトリからオペレータのインストール:

$ helm install weblogic-operator weblogic-operator/weblogic-operator

オペレータのアップグレード

オペレータ3.0.0では「下位互換性なし」の変更が導入されているため、helm upgradeを使用して2.6.0オペレータを3.xオペレータにアップグレードすることはできません。 かわりに、2.6.0オペレータを削除してから3.xオペレータをインストールする必要があります。

2.6.0オペレータを削除しても、ドメインCustomResourceDefinition (CRD)には「影響せず」、またすでに実行されているWebLogic Serverインスタンスは「停止しません」。

3.xオペレータがインストールされると、2.6.0オペレータによって作成された実行中のWebLogic Serverインスタンスが自動的にロールされます。 このローリング再起動では、他のローリング再起動と同様に、WebLogicクラスタの可用性保証(クラスタ・メンバーのみ)が保持されます。

2.6.0オペレータを削除するには:

$ helm delete weblogic-operator -n weblogic-operator-namespace

次に、前述の「インストール」の手順を使用して3.xオペレータをインストールします。

追加バージョンがリリースされると、3.xリリース・ファミリ内のアップグレード・オペレータに次の手順が適用されます。

オペレータをアップグレードするには、helm upgradeコマンドを使用します。 ローカル・マシン上のweblogic-kubernetes-operatorリポジトリが、アップグレード先のオペレータ・リリースであることを確認します。 オペレータをアップグレードする場合、helm upgradeコマンドでは新しいHelmチャートおよびイメージを指定する必要があります。 例えば: 

$ helm upgrade \
  --reuse-values \
  --set image=ghcr.io/oracle/weblogic-kubernetes-operator:3.1.3 \
  --namespace weblogic-operator-namespace \
  --wait \
  weblogic-operator \
  kubernetes/charts/weblogic-operator

オペレータのアンインストール

helm uninstallコマンドは、オペレータ・リリースとその関連リソースをKubernetesクラスタから削除するために使用します。 helm uninstallコマンドで使用されるリリース名は、helm installコマンドで使用されるリリース名と同じです(「Helmチャートのインストール」を参照)。 例えば: 

$ helm uninstall weblogic-operator -n weblogic-operator-namespace

Helmチャートのインストール前にオペレータのネームスペースが存在していなかった場合は、Helmによって作成されますが、helm uninstallによって削除されることはありません。

オペレータ・デプロイメントを削除した後、ドメイン・カスタム・リソース定義(CRD)も削除する必要があります:

$ kubectl delete customresourcedefinition domains.weblogic.oracle

ドメイン・カスタム・リソース定義は共有されています。 同じクラスタに他のオペレータがある場合は、CRDを削除しないでください。
