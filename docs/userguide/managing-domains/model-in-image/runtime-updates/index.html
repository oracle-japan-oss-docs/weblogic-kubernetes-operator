概要

実行中のModel in Imageドメインに構成変更を加え、その変更をWebLogic Serverポッドの再起動後も存続させる場合は、次のいずれかの方法を使用して既存のモデルを変更できます:

  • モデル・ファイル内のマクロによって参照されるシークレットまたは環境変数の変更。

  • モデル・ファイルを含む新しいWDT ConfigMapまたは更新されたWDT ConfigMapを指定し、ドメインYAMLファイルのconfiguration.model.configMapフィールドを使用してマップを参照します。

  • 新規または変更されたモデル・ファイルを使用した新規イメージの指定。

変更を適用した後、ドメインYAMLファイルのimageまたはrestartVersion属性を変更することで、変更を適用して実行中のドメインに伝播するようにオペレータに指示できます。

注意事項

  • 「サポート対象およびサポート対象外の更新」を確認します。

  • イメージまたはWDT ConfigMapで複数のモデル・ファイルを指定した場合、それらがロードおよびマージされる順序は、「モデル・ファイルの命名およびロード順序」で説明されているとおりに決定されます。

  • WDT Discover DomainツールとCompare Domainツールを使用すると、モデル・ファイルの更新を簡単に生成できます。 「WDT Discover DomainツールとCompare Modelツールの使用」を参照してください。

  • restartVersionを変更する簡単な方法は、「ドメインrestartVersionの変更」を参照してください。

  • 実行中のドメインにデータ・ソースを追加するサンプルは、「データ・ソースの追加例」を参照してください。

  • モデル・ファイルの構文の詳細は、WebLogic Deploy Toolのドキュメントおよび「モデル・ファイル」を参照してください。

  • イントロスペクタ・ジョブで障害が報告された場合は、デバッグのアドバイスについて「デバッグ」を参照してください。

よくある質問

モデル・ファイルを使用して更新を指定する必要があるのはなぜですか。

Domain in Imageと同様に、WebLogic Server管理コンソールまたはWLSTスクリプトを使用してModel in Imageドメインの直接実行時WebLogic構成を更新すると、更新は一時的になります。 これは、ドメイン・ホームがポッドの再起動後も存続しないイメージ・ディレクトリに格納されているためです。

Model in Imageの更新は実行時にどのように機能しますか。

ドメインのrestartVersionまたはimage属性を変更すると、オペレータはドメインのイントロスペクタ・ジョブを再実行します。 このジョブは、すべてのシークレットおよび環境変数をリロードし、すべてのモデル・ファイルをマージして、新しいドメイン・ホームを生成します。 ジョブが成功すると、オペレータはDOMAIN_UID-weblogic-domain-introspect-cmという名前のConfigMapを使用して、更新されたドメイン・ホームをポッドで使用できるようにします。 最後に、オペレータはドメイン内で実行中の各WebLogic Serverポッドをロール(再起動)して、新しい構成をロードできるようにします。 ドメイン・ロールは、ドメインの管理サーバーの再起動から始まり、ドメイン内の各管理対象サーバーの再起動に進みます。

かわりに、カスタム構成オーバーライドを使用して更新を実行できますか。

いいえ。カスタム構成オーバーライドは、「構成のオーバーライド」で説明されているように、ドメインYAMLファイルconfiguration.overridesConfigMapを使用して指定されたWebLogic構成オーバーライドであり、Model in Imageとの組合せではサポートされません。 カスタム・オーバーライドが指定されている場合、Model in Imageはエラーを生成します。 モデル・ファイル、シークレットまたはモデル・イメージの更新は、カスタム構成のオーバーライドの更新よりも単純で柔軟性が高いため、これは考慮しないでください。 構成のオーバーライドとは異なり、モデル・ファイルの更新の構文は、最初にモデル・ファイルを指定する構文と完全に一致します。

サポート対象およびサポート対象外の更新

サポートされていない更新を適用した場合、予期される動作は未定義です。 サポートされていない更新を行う必要があり、回避策がドキュメント化されていない場合は、変更を行う前にドメインを完全に停止します。

サポートされる更新

特に次の「サポート対象外」として記載されている領域を参照する場合を除き、次の更新はサポートされます:

  • 新しいWebLogicクラスタまたはスタンドアロン・サーバーを追加できます。

  • 対応するモデルYAMLファイル・スニペットを親Bean階層とともに指定することで、新しいMBeansまたはリソースを追加できます。 「データ・ソースの追加例」を参照してください。

  • YAMLファイル・スニペットと、既存のMBeanおよび属性を参照する親Bean階層を指定することで、MBean属性を変更または追加できます。 たとえば、mynewdatasourceという名前のデータ・ソースの最大容量を追加または変更するには: 

    resources:
  JDBCSystemResource:
    mynewdatasource:
      JdbcResource:
        JDBCConnectionPoolParams:
          MaxCapacity: 5

    詳細は、WebLogic Deploy Toolingドキュメントの「複数のモデルの使用」に関する項を参照してください。

  • モデル・マクロが参照するシークレット(@@SECRET:secretname:secretkey@@構文を使用するマクロ)を変更または追加できます。 たとえば、データベースのパスワード・シークレットを変更できます。

  • モデル・マクロが参照する環境変数(@@ENV:myenvvar@@構文を使用するマクロ)を変更または追加できます。

  • 名前付きのMBean、アプリケーション・デプロイメントまたはリソースを削除するには、名前の直前に感嘆符(!)を付けてモデル・ファイルを指定します。 たとえば、mynewdatasourceという名前のデータ・ソースがモデルに定義されている場合、データ・ソースを定義するモデル・ファイルの後にロードされる小さいモデル・ファイルを指定することで、そのデータ・ソースを削除できます。この場合、小さいモデル・ファイルは次のようになります: 

    resources:
  JDBCSystemResource:
    !mynewdatasource:

    詳細は、WebLogic Deploying Toolingのドキュメントの「削除する名前付きMBeansの宣言」に関する項を参照してください。

サポートされない更新

次の更新は「サポート対象外」です。 サポートされていない更新を行う必要があり、回避策がドキュメント化されていない場合は、変更を行う前にドメインを完全に停止します。

  • モデル・ファイルですでに指定されている属性をMBeanから直接削除する方法はありません。 これを回避するには、2つのモデル・ファイルを使用: 削除する属性の親である名前付きBean/リソースを削除するモデル・ファイルを追加し、削除する属性なしで名前付きbean/リソースを完全に定義する最初のモデル・ファイルの後にロードされる別のモデル・ファイルを追加します。

  • 属性のMBean名を直接変更する方法はありません。 かわりに、前述のように!構文を使用して名前付きMBeanを削除し、新しい構文を置換として追加できます。

  • ドメイン名は実行時に変更できません。

  • 既存のWebLogicクラスタのトポロジは変更できません。 具体的には、次のものには実行時更新を適用しないでください:

    • 動的クラスタ・サイズ
    • クラスタへのWebLogic Serversの追加または削除

  • 既存のクラスタまたはサーバーのネットワーク・リスニング・アドレス、ポート、プロトコルおよび有効な構成を実行時に変更、追加または削除することはできません。

    具体的には、次のものには実行時更新を適用しないでください:

    • デフォルト、SSL、管理チャネルEnabled、リスニング・アドレスまたはポート。
    • ネットワーク・アクセス・ポイント(カスタム・チャネル)Enabled、リスニング・アドレス、プロトコルまたはポート。

    ネットワーク・アクセス・ポイントのpublicまたはexternalのアドレスとポートをオーバーライドできます。 JMX (MBean)またはオンラインWLSTへの外部アクセスでは、ネットワーク・アクセス・ポイントの内部ポートと外部ポートが一致している必要があります(JMS、RMIまたはEJBへの外部T3またはHTTPトンネリング・アクセスでは、ポートの一致は必要ありません)。

    セキュリティ上の考慮事項により、T3またはRMIプロトコルをクラスタの外部に公開しないことを強くお薦めします。

  • spec.logHome、spec.logHomeEnabledまたはspec.httpAccessLogInLogHome属性を使用して同じMBeansをオーバーライドするようにドメイン・リソースが構成されている場合、実行時にMBeanのサーバーおよびドメイン・ログ関連の設定を変更、追加または削除することはできません。

  • 「ユーザー、グループ、ロール」および「資格証明マッピング」の組込みLDAPセキュリティ・エントリは変更できません。 たとえば、デフォルト・セキュリティ・レルムにユーザーを追加することはできません。 これらの種類の更新を行う必要がある場合は、変更を行う前にドメインを完全に停止するか、「外部セキュリティ・プロバイダ」に切り替えます。

  • 回避策が記載されていないかぎり、このリリースのModel in Imageでサポートされて「いない」実行時更新構成のタイプを次にまとめます:

    • 既存のWebLogicクラスタのドメイン・トポロジ。 具体的には、次のようになります。
      • 動的クラスタ・サイズ
      • クラスタへのWebLogic Serversの追加または削除
    • 既存のWebLogicクラスタまたはサーバーのデフォルトおよびカスタムのネットワーク・チャネル構成。 具体的には、次のようになります。
      • 既存のサーバーのネットワーク・アクセス・ポイント(カスタム・チャネル)の追加または削除
      • デフォルト、SSL、管理またはカスタム・チャネル、Enabled、リスニング・アドレス、プロトコルまたはポートの変更
    • ノード・マネージャ関連の構成
    • 既存のMBean名の変更
    • MBean属性の削除
    • 組込みLDAPエントリ

ドメインrestartVersionの変更

「概要」で説明したように、実行中のドメインに構成変更を適用するようオペレータに指示するには、ドメインrestartVersionを変更する方法があります。 これを行う一般的な方法は次のとおりです:

  • kubectl edit -n MY_NAMESPACE domain MY_DOMAINUIDを使用して、restartVersionを対話形式で変更できます。

  • ドメインのリソース・ファイルがある場合は、このファイルを変更して、ファイルに対してkubectl apply -fをコールできます。

  • Kubernetesのgetおよびpatchコマンドを使用できます。 次に、ネームスペースを最初のパラメータとして取り(デフォルトはsample-domain1-ns)、domainUIDを2番目のパラメータとして取る自動化スクリプトの例を示します(デフォルトはsample-domain1): 

    #!/bin/bash
NAMESPACE=${1:-sample-domain1-ns}
DOMAINUID=${2:-sample-domain1}
currentRV=$(kubectl -n ${NAMESPACE} get domain ${DOMAINUID} -o=jsonpath='{.spec.restartVersion}')
if [ $? = 0 ]; then
  # we enter here only if the previous command succeeded

  nextRV=$((currentRV + 1))

  echo "@@ Info: Rolling domain '${DOMAINUID}' in namespace '${NAMESPACE}' from restartVersion='${currentRV}' to restartVersion='${nextRV}'."

  kubectl -n ${NAMESPACE} patch domain ${DOMAINUID} --type='json' \
    -p='[{"op": "replace", "path": "/spec/restartVersion", "value": "'${nextRV}'" }]'
fi

WDT Discover DomainツールとCompare Modelツールの使用

WebLogic Deploy Tooling 「ドメインの検出ツール」は既存のドメイン・ホームからモデル・ファイルを生成し、その「モデルの比較ツール」は2つのドメイン・モデルを比較して、最初のドメインを2つ目のドメインに更新するためのYAMLファイルを生成します。 これらのツールを組み合せて使用すると、既存のモデルを更新するために指定する必要があるモデル・ファイルの内容を特定できます。

たとえば、WDTを/u01/wdt/weblogic-deployにインストールし、ドメイン・タイプがWLSであるとします:


# (1) Run discover for your existing domain home.

$ /u01/wdt/weblogic-deploy/bin/discoverDomain.sh \
  -oracle_home $ORACLE_HOME \
  -domain_home $DOMAIN_HOME \
  -domain_type WLS \
  -archive_file old.zip \
  -model_file old.yaml \
  -variable_file old.properties

# (2) Now make some WebLogic config changes using the console or WLST.

# (3) Run discover for your changed domain home.

$ /u01/wdt/weblogic-deploy/bin/discoverDomain.sh \
  -oracle_home $ORACLE_HOME \
  -domain_home $DOMAIN_HOME \
  -domain_type WLS \
  -archive_file new.zip \
  -model_file new.yaml \
  -variable_file new.properties

# (4) Compare your old and new yaml using diff

$ diff new.yaml old.yaml

# (5) Compare your old and new yaml using compareDomain to generate
#     the YAML update file you can use for transforming the old to new.

# /u01/wdt/weblogic-deploy/bin/compareModel.sh \
  -oracle_home $ORACLE_HOME \
  -output_dir /tmp \
  -variable_file old.properties \
  old.yaml \
  new.yaml

# (6) The compareDomain will generate these files:
#      /tmp/diffed_model.json
#      /tmp/diffed_model.yaml, and
#      /tmp/compare_model_stdout

ノート: ドメイン・タイプがWLSでない場合は、前述のdiscoverDomain.shコマンドでドメイン・タイプをJRFまたはRestrictedJRFに変更してください。
