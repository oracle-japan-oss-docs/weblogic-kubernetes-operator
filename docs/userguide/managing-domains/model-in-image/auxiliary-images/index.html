導入

補助イメージは、Model in Imageモデル・ファイル、アプリケーション・アーカイブ・ファイル、WebLogic Deploy Toolingのインストール・ファイル、またはその他のタイプのファイルをポッドに含める代替アプローチです。 この機能により、domain.spec.imageで指定されたイメージにこれらのファイルを指定する必要がなくなります。

かわりに次のようにします。

  • ドメイン・リソースのdomain.spec.imageは、WebLogicインストールおよびJavaインストールのみを含める必要があるベース・イメージを直接参照します。
  • ドメイン・リソースの補助イメージ関連フィールドは、目的のModel in Imageファイルを含む1つ以上の小さいイメージを参照します。
  • ドメイン・リソースのdomain.spec.configuration.model.wdtInstallHomeおよびdomain.spec.configuration.model.modelHomeフィールドは、小さいイメージからのファイルを含むディレクトリを参照するように設定されます。

Model In Imageドメインにおける補助イメージの利点は次のとおりです:

  • WDTインストール、アプリケーション・アーカイブまたはモデル・アーティファクトをイメージに含める必要なく、WebLogicインストール・イメージを使用またはパッチ適用します。
  • 1つのWebLogicインストール・イメージを、特定のイメージで提供されている複数の異なるモデル構成と共有します。
  • WebLogicインストールを含まない特定のイメージを使用して、モデル・ファイル、アプリケーション・アーカイブおよびWebLogic Deploy Tooling実行ファイルを配布または更新します。

補助イメージは、内部的にKubernetes emptyDirボリュームおよびKubernetes initコンテナを使用して、追加のイメージからファイルを共有します。

参照

  • kubectl explain domain.spec.auxiliaryImageVolumesおよびkubectl explain domain.spec.serverPod.auxiliaryImagesコマンドを実行します。

  • ドメイン・リソースschemaおよびdocumentationのspec.auxiliaryImageVolumesおよびserverPod.auxiliaryImagesセクションを参照してください。

構成

この項では、Model in Imageユースケースの一般的な補助イメージ構成について説明します。

補助ボリュームとパス

ドメイン・リソースdomain.spec.auxiliaryImageVolumesの補助イメージ・ボリュームは、mountPath、nameおよび「追加のオプション・フィールド」を定義します。 mountPathフィールドは補助イメージ内のディレクトリのロケーションであり、メイン・ポッド・コンテナのロケーションでもあります(補助イメージ・ディレクトリの再帰的コピーが自動的に含まれます)。 nameフィールドは任意で、「補助イメージ」構成を使用して個別に定義される1つ以上の補助イメージによって参照されます。

例えば:

  spec:
    auxiliaryImageVolumes:
    - name: auxiliaryImageVolume1
      mountPath: /auxiliary

補助イメージ

1つ以上の補助イメージをドメイン・リソースserverPod.auxiliaryImage配列で構成できます。 各配列エントリでは、imageおよびvolumeを定義する必要があります。ここで、imageは補助イメージの名前、volumeは前述の「補助イメージ・ボリューム」の名前です。 オプションで、imagePullPolicyを指定することもできます。imageが:latestで終了する場合、IfNotPresentにデフォルト設定されます。それ以外の場合もあります。 また、オプションで、ポッド起動時に補助イメージのファイルを補助イメージ・ボリュームにマージ(コピー)するために使用されるコマンドをカスタマイズできます(これはめったに必要ありません。例については、「マージではなく置換の実行」を参照してください)。 各フィールドの詳細は、schemaを参照してください。

serverPodは、ドメイン内のすべてのポッド、イントロスペクタ・ジョブのポッド、特定のWebLogicクラスタのスコープ、または特定のWebLogic Serverポッドのスコープに適用されるドメイン・スコープで定義できます。 通常、ドメイン・スコープは、Model in Imageユースケースに最も適しています。次に例を示します: 

spec:
  serverPod:
    auxiliaryImages:
    - image: model-in-image:v1
      imagePullPolicy: IfNotPresent
      volume: auxiliaryImageVolume1

補助イメージのプルにイメージ・プル・シークレットが必要な場合は、domain.spec.imagePullSecretsを使用してシークレットを参照する必要があります。

Model in Imageパス

Model In Imageの補助イメージ・ユースケースでは、domain.spec.configuration.model.modelHomeおよびdomain.spec.configuration.model.wdtInstallHome属性を構成して、ドメインのWebLogic Deploy Tooling (WDT)モデル・ファイルとドメインのWDTインストールのロケーションを指定する必要があります。 これらはそれぞれ/u01/wdt/modelsおよび/u01/wdt/weblogic-deployにデフォルト設定され、domain.spec.auxiliaryImageVolumes.mountPathでディレクトリを指定するように変更する必要があります。 例えば: 

  configuration:
    model:
      modelHome: "/auxiliary/models"
      wdtInstallHome: "/auxiliary/weblogic-deploy"

複数の補助イメージが同じwdtInstallHomeパスに異なるバージョンのWebLogic Deploy Toolインストールを指定する場合は、新しいバージョンがマージではなく古いバージョンを完全に置き換えるようにすることをお薦めします。 「マージではなく置換の実行」を参照してください。

オーダーのマージ

すべての同じボリュームを参照する複数の補助イメージからのファイルのマージ順序を制御する必要がある場合、または補助イメージからそのボリュームにファイルをコピーするために使用するコマンドをカスタマイズする場合は、このセクションを参照してください。 コマンドのカスタマイズを使用して、マージ動作ではなく動作を強制的に置き換えることができます。

予想マージ順序

デフォルトでは、同じボリュームを共有する複数の補助イメージからのファイルがマージされます。 具体的には、次のようになります。

  • マージ内の後のイメージからのファイルは、以前のイメージからの同じ名前付きファイルを上書きします。

  • 複数のイメージから重複するディレクトリの内容が結合されます。

同じ補助イメージ・ボリュームを共有する補助イメージの予想マージ順序は次のとおりです:

  • 異なるserverPodスコープで補助イメージを指定し、すべて同じボリュームを共有する場合、ドメイン・スコープのファイルが最初にマージされ、クラスタ・スコープが2番目になり、サーバーのスコープが最後になります。

  • 同じスコープで複数の補助イメージを指定し、それらがすべて同じボリュームを共有している場合、ファイルはserverPod.auxiliaryImagesの下に表示される順序でマージされます。

ノート: マージの結果によって、補助イメージ・ボリューム内のdomain.spec.configuration.model.ModelHomeモデル・ファイルが2つ以上異なる場合、モデル・ファイルのロード順序は、モデル・ファイルの命名およびロード順序を参照してください。

マージではなく置換の実行

複数の補助イメージが同じボリュームを共有しており、2つのディレクトリを結合するのではなく、新しいイメージからの特定の同じ名前のディレクトリが前のイメージのディレクトリを完全に置き換えるようにする場合は、2番目のイメージが強制的に置き換えるために共有ボリュームを移入するために使用するコマンドをカスタマイズできます。

たとえば、2番目のイメージのserverPod.auxiliaryImage.commandフィールドをカスタマイズして、前のイメージからすでにコピーされたディレクトリを削除してから、通常のコピーを実行できます:

...
spec:
  ...
  auxiliaryImageVolumes:
  - name: aivolume
    mountPath: /auxiliary
  ...
  serverPod:
    auxiliaryImages:
    - image: domain-image-A:v1
      volume: aivolume
    - image: domain-image-B:v1
      volume: aivolume
      # the following command replaces 'mydir' instead of merging it:
      command: 'rm -fr $AUXILIARY_IMAGE_TARGET_PATH/mydir; cp -R $AUXILIARY_IMAGE_PATH/* $AUXILIARY_IMAGE_TARGET_PATH'

マージ順序の例

補助イメージがドメイン、WebLogicクラスタおよびサーバー・スコープで定義されている場合(myserverはmyclusterの一部です):

...
spec:
  auxiliaryImageVolumes:
  - name: aivolume
    mountPath: /auxiliary
  serverPod:
    auxiliaryImages:
    - image: domain-image-A:v1
      volume: aivolume
    - image: domain-image-B:v1
      volume: aivolume
  cluster:
  - name: mycluster
    serverPod:
      auxiliaryImages:
      - image: cl-image-A:v1
        volume: aivolume
      - image: cl-image-B:v1
        volume: aivolume
  managedServers:
  - serverName: "myserver"
    serverPod:
      auxiliaryImages:
      - image: ms-image-A:v1
        volume: aivolume

その後、イメージからのファイルは、次の順序で共有aivolumeボリューム/auxiliaryマウント・パスにマージされます:

domain-image-A:v1 (first)
domain-image-B:v1
cl-image-A:v1
cl-image-B:v1
ms-image-A:v1 (last)

サンプル

このサンプルは、補助イメージを使用してドメインのWDTモデル・ファイル、アプリケーション・アーカイブZIPファイルおよびWDTインストールを小さい個別のコンテナ・イメージで提供するModel in Imageドメインのデプロイを示しています。

ステップ1: 前提条件

  • まず、Model in Image 「初期ユース・ケース・サンプル」のすべてのステップに従います。

    これにより、次のようになります:

    • ドメインのオペレータおよびネームスペースを設定します。
    • WebLogic Deploy Tool ZIPインストールをダウンロードします。
    • ドメインの「なしで」補助イメージをデプロイします。

  • 次に、ドメインをシャットダウンし、ポッドが終了するまで待ちます。

    • wl-pod-wait.shスクリプトを使用して待機できます。
    • たとえば、/tmp/mii-sampleを作業ディレクトリとして設定するとします: 
      $ kubectl delete domain sample-domain1 -n sample-domain1-ns
$ /tmp/mii-sample/utils/wl-pod-wait.sh -p 0

ステップ2: 補助イメージの作成

次のステップに従って、Model In Imageモデル・ファイル、アプリケーション・アーカイブおよびWDTインストール・ファイルを含む補助イメージを作成します:

  1. モデルZIPアプリケーション・アーカイブを作成し、モデルYAMLファイルおよびモデル・プロパティ・ファイルが初期ユースケース用にすでに配置されている同じディレクトリに配置します:

    $ rm -f /tmp/mii-sample/model-images/model-in-image__WLS-AI-v1/archive.zip
$ cd /tmp/mii-sample/archives/archive-v1
$ zip -r /tmp/mii-sample/model-images/model-in-image__WLS-AI-v1/archive.zip wlsdeploy

    このサンプルを以前実行したアーカイブZIPの古いバージョンがある場合、rm -fコマンドが含まれます。

  2. 補助イメージのファイルをステージングするための一時ディレクトリを作成し、cdをこのディレクトリに作成します:

    $ mkdir -p /tmp/mii-sample/ai-image/WLS-AI-v1
$ cd /tmp/mii-sample/ai-image/WLS-AI-v1

    このディレクトリWLS-AI-v1をコールして、補助イメージに使用するイメージ・バージョン・タグに対応します。

  3. WDTをステージング・ディレクトリにインストールし、UNIX環境で使用されていないweblogic-deploy/bin/*.cmdファイルを削除します:

    $ unzip /tmp/mii-sample/model-images/weblogic-deploy.zip -d .
$ rm ./weblogic-deploy/bin/*.cmd

    後のステップでは、このWDTインストール・ディレクトリを参照するドメイン・リソースdomain.spec.configuration.model.wdtInstallHome属性を指定します。

    weblogic-deploy.zipファイルがない場合、前提条件のステップをスキップしました。

  4. ステージング・ディレクトリにmodelsディレクトリを作成し、モデルYAMLファイル、プロパティおよびアーカイブをコピーします:

    $ mkdir ./models
$ cp /tmp/mii-sample/model-images/model-in-image__WLS-AI-v1/model.10.yaml ./models
$ cp /tmp/mii-sample/model-images/model-in-image__WLS-AI-v1/model.10.properties ./models
$ cp /tmp/mii-sample/model-images/model-in-image__WLS-AI-v1/archive.zip ./models

    後のステップでは、このディレクトリを参照するドメイン・リソースのdomain.spec.configuration.model.modelHome属性を指定します。

  5. /tmp/mii-sample/ai-docker-file/Dockerfileを使用してdocker buildを実行し、小さいbusyboxイメージをベース・イメージとして使用して補助イメージを作成します。

    $ docker build -f /tmp/mii-sample/ai-docker-file/Dockerfile \
  --build-arg AUXILIARY_IMAGE_PATH=/auxiliary \
  --tag model-in-image:WLS-AI-v1 .

    各ビルド引数については、./Dockerfileを参照してください。

    「Dockerfileを表示するには、ここをクリックします。」 
    # Copyright (c) 2021, Oracle and/or its affiliates.
# Licensed under the Universal Permissive License v 1.0 as shown at https://oss.oracle.com/licenses/upl.

# This is a sample Dockerfile for supplying Model in Image model files
# and a WDT installation in a small separate auxiliary
# image. This is an alternative to supplying the files directly
# in the domain resource `domain.spec.image` image.

# AUXILIARY_IMAGE_PATH arg:
#   Parent location for Model in Image model and WDT installation files.
#   Must match domain resource 'domain.spec.auxiliaryImageVolumes.mountPath'
#   For model-in-image, the following two domain resource attributes can
#   be a directory in the mount path:
#     1) 'domain.spec.configuration.model.modelHome'
#     2) 'domain.spec.configuration.model.wdtInstallHome'
#   Default '/auxiliary'.
#

FROM busybox
ARG AUXILIARY_IMAGE_PATH=/auxiliary
ARG USER=oracle
ARG USERID=1000
ARG GROUP=root
ENV AUXILIARY_IMAGE_PATH=${AUXILIARY_IMAGE_PATH}
RUN adduser -D -u ${USERID} -G $GROUP $USER
COPY ./ ${AUXILIARY_IMAGE_PATH}/
RUN chown -R $USER:$GROUP ${AUXILIARY_IMAGE_PATH}
USER $USER

  6. イメージの作成後、WDT実行可能ファイルは/auxiliary/weblogic-deployにあり、WDTモデル、プロパティおよびアーカイブ・ファイルは/auxiliary/modelsにあります。 Dockerイメージでlsを実行して、次のことを確認できます:

  drwxr-xr-x    1 oracle   root          4096 Jun  1 21:53 models
  drwxr-xr-x    1 oracle   root          4096 May 26 22:29 weblogic-deploy

$ docker run -it --rm model-in-image:WLS-AI-v1 ls -l /auxiliary/models
  total 16
  -rw-rw-r--    1 oracle   root          5112 Jun  1 21:52 archive.zip
  -rw-rw-r--    1 oracle   root           173 Jun  1 21:59 model.10.properties
  -rw-rw-r--    1 oracle   root          1515 Jun  1 21:59 model.10.yaml

$ docker run -it --rm model-in-image:WLS-AI-v1 ls -l /auxiliary/weblogic-deploy
  total 28
  -rw-r-----    1 oracle   root          4673 Oct 22  2019 LICENSE.txt
  -rw-r-----    1 oracle   root            30 May 25 11:40 VERSION.txt
  drwxr-x---    1 oracle   root          4096 May 26 22:29 bin
  drwxr-x---    1 oracle   root          4096 May 25 11:40 etc
  drwxr-x---    1 oracle   root          4096 May 25 11:40 lib
  drwxr-x---    1 oracle   root          4096 Jan 22  2019 samples
