WebLogic Kubernetes Operator

オペレータをデプロイし、Model in Imageドメインの目的のネームスペースをモニタリングしていることを確認します。 「オペレータの管理」および「クイック・スタート」を参照してください。

WebLogic Serverイメージ

Model in Imageには、WebLogic Serverインストールを含むイメージが必要です。

  • WebLogic Server 12.2.1.4以降のOracle Container Registryの事前作成されたベース・イメージ(WLSドメインの場合はcontainer-registry.oracle.com/middleware/weblogic:12.2.1.4、JRFドメインの場合はcontainer-registry.oracle.com/middleware/fmw-infrastructure:12.2.1.4など)から開始できます。

    ベース・イメージの例はGAイメージで、パブリック・インターネットから環境を使用できないデモおよび開発目的「のみ」に適しています。これらは「本番で使用できません」。 本番では、常にOCRのCPU (パッチ適用済)イメージを使用するか、WebLogic Image Tool (WIT)と--recommendedPatchesオプションを使用してイメージを作成する必要があります。 詳細は、「Oracle WebLogic Serverの本番環境の保護」の「最新のパッチと更新の適用」を参照してください。

    WLSドメインとJRFドメインの両方に対するこのアプローチの例は、Model in Imageサンプルを参照してください。 Oracle Container Registryにログインしてイメージのライセンス契約に同意する方法の詳細は(Oracle Container Registryイメージのプルを許可するために必要)、この「ドキュメント」を参照してください。

  • または、「パッチを適用したカスタム・イメージの作成」の説明に従って独自のベース・イメージを手動で構築できます。 これは、ベース・イメージに追加のパッチを含める場合に便利です。 12.2.1.3イメージにもパッチ29135930が含まれている必要があることに注意してください(事前作成イメージにはすでにこのパッチが含まれています)。

ディレクトリ構造

Model in Imageでは、(オプション) WDTモデル・アーティファクトおよび(必須) WDTバイナリのポッドに次のディレクトリ構造が必要です:

ドメイン・リソース属性 デフォルト・ディレクトリ 目次
domain.spec.configuration.model.modelHome /u01/wdt/models ゼロ以上のモデル.yaml、.properties、または.zipファイルをアーカイブします。
domain.spec.configuration.model.wdtInstallHome /u01/wdt/weblogic-deploy 解凍されたWDTインストール・バイナリ(必須)。

初期モデル・ファイルとWDTの指定

Model in Imageには、WebLogicインストール(「WebLogic Serverイメージ」を参照)と次のアクセスを含むイメージが最小限必要です:

  • domain.spec.configuration.model.wdtInstallHomeのWDTインストール。
  • domain.spec.configuration.model.modelHomeディレクトリまたは「オプションのWDTモデルConfigMap」にドメインを構成する1つ以上のWDTモデル.yamlファイル。
  • domain.spec.configuration.model.modelHomeディレクトリまたは「オプションのWDTモデルConfigMap」に0個以上のWDTモデル.propertiesファイルがあります。
  • domain.spec.configuration.model.modelHomeディレクトリに0個以上のWDTモデル・アプリケーション.zipアーカイブがあります。 アプリケーション・アーカイブは「オプションのWDTモデルConfigMap」でサポートされていないため、モデル・ホームでアーカイブを指定する必要があります。

Model in Image WDTアーティファクトを指定するメソッドは複数あります:

  • メイン・イメージに含める: ドメイン・リソースdomain.spec.imageのアーティファクトをベース・イメージ(ベース・イメージにWebLogicインストールが含まれている)上のレイヤーとしてdomain.spec.configuration.model.modelHomeおよびdomain.spec.configuration.model.wdtInstallHomeディレクトリに含めることができます。

    次のいずれかのメソッドを使用します。

    • 「手動イメージ作成」は、Dockerコマンドを使用して、前の表で説明したWDTアーティファクトを新しいイメージにレイヤー化します。
    • WebLogic Image Tool (WIT)には、WDTモデル・ファイルのレイヤー化、WDTバイナリ、WebLogic ServerバイナリおよびWebLogic Serverパッチをイメージ内に配置するための組込みオプションがあります。 Model in Imageサンプルでは、WITアプローチを使用します。

  • 補助イメージの使用: 補助イメージを使用して、目的のファイルを含む1つ以上の小さいイメージを作成します。 これにより、各ポッドのファイル・システムのconfiguration.model.modelHomeまたはconfiguration.model.wdtInstallHomeのロケーションに、小さいイメージからファイルが自動的にコピーされます。

  • 永続ボリューム要求(PVC)の使用: このメソッドは、高度なユースケース専用です。 WDTモデルYAML、変数またはアーカイブ・ファイルを「永続ボリューム要求」に指定し、configuration.model.modelHomeおよびconfiguration.model.wdtInstallHomeをPVCのマウント・ロケーション内の対応するディレクトリに変更します。

  • WDTモデルConfigMapの使用: WDTモデルYAMLおよび.propertiesファイルには、WDTモデルConfigMap(オプション)を使用します。 これは、前述のいずれかのメソッドと組み合せることができます。ほとんどの場合、これらのメソッドのいずれかによって提供されるモデルへのランタイム更新を容易にするために使用されます。

モデル・ファイルの構文の詳細は、「モデル・ファイル」を参照してください。

WDTモデルConfigMap(オプション)

イメージにすでに指定されているもの以外のモデル.yamlおよび.propertiesファイルを定義するWDTモデルConfigMapを作成し、ドメインYAMLファイルのconfiguration.model.configMap属性を使用してこのConfigMapを参照できます。 指定されたイメージがすでにモデルを完全に定義している場合、これはオプションです。

WDTモデルのConfigMapファイルは、ドメイン・ホームが作成される前に、実行時にイメージに定義されたWDTファイルとマージされます。 ConfigMapファイルは、イメージ内で指定したモデル構成に対して追加、削除または変更できます。

たとえば、追加の.yamlおよび.propertiesファイルを/home/acmeuser/wdtoverrideというディレクトリに配置し、次のコマンドを実行します:

$ kubectl -n MY-DOMAIN-NAMESPACE \
  create configmap MY-DOMAINUID-my-wdt-config-map \
  --from-file /home/acmeuser/wdtoverride

$ kubectl -n MY-DOMAIN-NAMESPACE \
  label  configmap MY-DOMAINUID-my-wdt-config-map \
  weblogic.domainUID=MY-DOMAINUID

モデル・ファイルの構文およびロード順序については、「モデル・ファイル」を参照してください。WDTモデルConfigMapsを使用して実行中のドメインのモデル構成を更新する方法については、「ランタイムの更新」を参照してください。

必要なランタイム暗号化シークレット

Model in Imageには、セキュアなpasswordキーを使用したランタイム暗号化シークレットが必要です。 このシークレットは、モデルおよびドメイン・ホーム・アーティファクトをランタイムConfigMapまたはログに追加する前に暗号化するためにオペレータによって使用されます。 ドメインを完全に停止した後はいつでもpasswordを安全に変更できますが、実行中のドメインの存続期間中は同じままにする必要があります。 作成するランタイム暗号化シークレットには任意の名前を付けることができますが、クリーンアップ・スクリプトがシークレットを見つけて削除できるように、シークレットにドメインUIDの名前を付けてラベルを付けることをお薦めします。

ノート: ランタイム暗号化パスワードは共有する必要はなく、ドメインの存続期間中のみ存在する必要があるため、パスワード・ジェネレータを使用できます。

例:

$ kubectl -n MY-DOMAIN-NAMESPACE \
  create secret generic MY-DOMAINUID-runtime-encrypt-secret \
  --from-literal=password=welcome1

$ kubectl -n MY-DOMAIN-NAMESPACE \
  label secret MY-DOMAINUID-runtime-encrypt-secret \
  weblogic.domainUID=MY-DOMAINUID

対応するドメインYAMLファイル・スニペット:

configuration:
  model:
    runtimeEncryptionSecret: MY-DOMAINUID-runtime-encrypt-secret

モデル・マクロのシークレット

モデル・ファイル内のマクロの必要に応じて、追加のシークレットを作成します。 たとえば、シークレットを参照するモデル内の@@SECRETマクロを使用してアクセスされるデータベースURLおよび資格証明を格納できます。 モデル・マクロの説明については、「モデル・ファイル」を参照してください。

ドメイン・フィールド

次のドメイン・フィールドは、Model in Imageドメインに固有です。

ドメイン・リソース属性 ノート
domainHomeSourceType 必須。 FromModelに設定します。
domainHome イメージ内の空または存在しないディレクトリを参照する必要があります。 永続ボリュームのマウント・パスは含めないでください。 ポッドが再起動するたびに、Model in ImageによってWebLogic Serverポッドのドメイン・ホームが再作成されることに注意してください。
configuration.model.configMap オプション。 「WDTモデルConfigMap(オプション)」に従って追加のモデルをConfigMapに格納した場合に構成します。
configuration.secrets オプション。 カスタムKubernetesシークレットを参照するマクロがイメージまたはConfigMapモデルに含まれている場合は、この配列を構成します。 たとえば、マクロがシークレットmy-secretおよびmy-other-secretに依存している場合は、[my-secret, my-other-secret]に設定します。
configuration.model.runtimeEncryptionSecret 必須。 すべてのModel in Imageドメインは、実行時暗号化シークレットを指定する必要があります。 「必要なランタイム暗号化シークレット」を参照してください。
configuration.model.domainType ドメインのタイプを設定します。 有効な値は、WLS、JRFおよびRestrictedJRFで、WLSがデフォルトです。 「WDTドメイン・タイプ」を参照してください。
configuration.model.runtimeEncryptionSecret 必須。 すべてのModel in Imageドメインは、実行時暗号化シークレットを指定する必要があります。 「必要なランタイム暗号化シークレット」を参照してください。
configuration.model.modelHome オプション。 モデルYAMLファイル、.propertiesファイルおよびアプリケーション.zipアーカイブを含むWDTモデル・ホームのロケーション。 デフォルトは/u01/wdt/modelsです。
configuration.model.wdtInstallHome オプション。 WDTインストールのロケーション。 デフォルトは/u01/wdt/weblogic-deployです。

ノート:

  • imageフィールドなど、すべてのドメイン・ホーム・ソース・タイプに共通する追加属性があります。 ドメイン・フィールドの完全なリストは、ドメイン・リソース「スキーマ」および「ドキュメンテーション」を参照してください。

  • JRFドメイン・タイプに固有の追加フィールドもあります。 詳細は、「JRFドメイン・タイプの要件」を参照してください。

  • Model in ImageドメインYAMLファイルの完全な例については、「Model in Imageサンプル」のオペレータのソース・ディレクトリkubernetes/samples/scripts/create-weblogic-domain/model-in-image/domain-resourcesを参照してください。 このディレクトリのWLSおよびJRFサブディレクトリは、configuration.model.domainTypeに対応します。

常に外部状態を使用

ドメイン・ホームのソース・タイプに関係なく、常にイメージの外部に状態を保持することをお薦めします。 これには、クラスタ・データベース・リース表、JMSおよびトランザクション・ストア、EJBタイマーなどが含まれます。 これにより、コンテナが破棄されたときにデータが失われることがなくなります。

組込みデータベース・サーバーの高可用性機能を利用するには、状態をデータベースに保持することをお薦めします。また、最短距離を除くすべてのサイトの障害時リカバリでは、データの統合およびレプリケートに常に単一のデータベース・サーバー(DataGuard)を使用する必要があります。

詳細は、次のドキュメントを参照してください。

  • 「Oracle WebLogic Serverのパフォーマンス・チューニング」の「JDBCストアのチューニング」。
  • 「WebLogic永続ストアの管理」の「JDBCストアの使用」。
  • 「Oracle WebLogic ServerのJMSリソースの管理」の「高可用性のベスト・プラクティス」。
  • 「Oracle WebLogic Serverクラスタの管理」の「リース」。

JRFドメイン・タイプの要件

この項は、JRFドメイン・タイプにのみ適用されます。 ドメイン・タイプがWLSまたはRestrictedJRFの場合はスキップします。

JRFドメインには、インフラストラクチャ・データベース、RCUを使用したこのデータベースの初期化、およびこのデータベースにアクセスするためのドメインの構成が必要です。 これらのステップはすべて、最初にドメインをデプロイする前に実行する必要があります。 ドメインを初めてデプロイすると、イントロスペクタ・ジョブによってデータベースのOPSSスキーマ表が初期化されます - 数分かかる場合があるプロセス。

さらに、再起動されたJRFドメインが、以前に作成されたインフラストラクチャ・データベースへの更新に安全にアクセスできるようにするには、元のドメインのウォレット・ファイルを実際どおりに安全に保存し、再起動されたドメインに、ドメインの以前の実行から取得したウォレット・ファイルを指定する必要があります。

JRFドメインYAMLファイルおよびモデルYAMLファイルの設定

Model in Image JRFドメインに必要なドメインYAMLファイルおよびモデルYAMLファイルの設定を次に示します:

  • configuration.model.domainTypeをJRFに設定します。

  • walletPasswordキーを定義するシークレットを参照するようにconfiguration.opss.walletPasswordSecretを設定します。 これは、ドメインのOPSSウォレット・ファイルの暗号化に使用されます。 JRFドメインの場合、これは必須フィールドです。

  • walletFileキーにドメインのOPSSウォレット・ファイルを含むシークレットを参照するようにconfiguration.opss.walletFileSecretを設定します。 これは、同じドメインの以前の起動からOPSSウォレット・ファイルがあることを前提としています。 再起動または移行されたドメインがデータベース情報にアクセスできるようにします。 JRFドメインの場合、これはオプションのフィールドですが、再起動または移行されたドメインでデータベース情報にアクセスする場合は、常に設定する必要があります。

  • configuration.introspectorJobActiveDeadlineSecondsイントロスペクション・ジョブのタイムアウトを600秒以上に設定します。 これはオプションのフィールドですが、ドメイン・ホームの作成にはJRFドメインの初回作成時にかなりの時間がかかり(ドメインのデータベース表の初期化が原因)、Model in Imageではイントロスペクション・ジョブを使用してドメイン・ホームが作成されるため必要です。

  • モデルにRCUDbInfoスタンザを定義します。 データベースにアクセスするには、データベース内のドメインのスキーマにアクセスするために必要な情報を使用して、モデルのdomainInfoスタンザでRCUDbInfoスタンザを定義する必要があります。 通常、この情報は、ドメインYAMLファイルのconfiguration.secretsフィールドでデプロイおよび参照するシークレットを使用して指定する必要があります。 RCUDbInfoスタンザの例を次に示します: 

    domainInfo:
    RCUDbInfo:
        rcu_prefix:          '@@SECRET:sample-domain1-rcu-access:rcu_prefix@@'
        rcu_schema_password: '@@SECRET:sample-domain1-rcu-access:rcu_schema_password@@'
        rcu_db_conn_string:  '@@SECRET:sample-domain1-rcu-access:rcu_db_conn_string@@'
JRFウォレットの保存およびリストア

JRFドメインのOPSSウォレット・パスワードおよびウォレット・ファイルは、必要に応じてリストアできるように保存することが重要です。 これにより、ドメインの再起動または移行がドメインのFMWインフラストラクチャ・データベースに引き続きアクセスできるようになります。

JRFドメインを初めてデプロイすると、ドメイン自体がRCUデータベース表に追加され、RCUデータベース内のドメインのデータへのアクセスを可能にするウォレット・ファイルもドメインのホーム・ディレクトリに作成されます。 このウォレットは、ドメインYAMLファイルのconfiguration.opss.walletPasswordSecretフィールドで参照されるシークレットを使用してドメインに指定するOPSSキー・パスワードを使用して暗号化されます。

Model in Imageによって起動されたドメインの場合、オペレータは新しいJRFドメインのドメイン・ホームからウォレット・ファイルをコピーし、ewallet.p12ファイル内のドメインのイントロスペクタ・ドメインConfigMapに格納します。 イントロスペクタ・ドメインConfigMapからこのウォレット・ファイルをエクスポートする方法を次に示します:

  • オプション1

    $ kubectl -n MY_DOMAIN_NAMESPACE \
  get configmap MY_DOMAIN_UID-weblogic-domain-introspect-cm \
  -o jsonpath='{.data.ewallet\.p12}' \
  > ewallet.p12

  • オプション2

    または、./kubernetes/samples/scripts/create-weblogic-domain/model-in-image/utils/opss-wallet.sh -sコマンドを使用してウォレット・ファイルをエクスポートできます(このスクリプトのコマンドライン引数およびデフォルトに-?を渡します)。

ウォレット・ファイルは、後で取得できる安全なロケーションに必ずバックアップしてください。 さらに、OPSSキーのパスワードを保存します。

ウォレットを再利用するには:

  • 元のドメインで指定したものと同じOPSSパスワードを含むwalletPasswordという名前のキーを使用してシークレットを作成します。 たとえば、パスワードがwelcome1であるとします: 
    $ kubectl -n MY_DOMAIN_NAMESPACE \
  create secret generic MY_DOMAIN_UID-my-opss-wallet-password-secret \
  --from-literal=walletPassword=welcome1

    $ kubectl -n MY_DOMAIN_NAMESPACE \
  label secret MY_DOMAIN_UID-my-opss-wallet-password-secret \
  weblogic.domainUID=sample-domain1
  • 前にエクスポートしたOPSSウォレット・ファイルを含むwalletFileという名前のキーを使用してシークレットを作成します。 たとえば、ファイルがewallet.p12であるとします: 
    $ kubectl -n MY_DOMAIN_NAMESPACE \
  create secret generic MY_DOMAIN_UID-my-opss-wallet-file-secret \
  --from-file=walletFile=ewallet.p12

    $ kubectl -n sample-domain1-ns \
  label secret MY_DOMAIN_UID-my-opss-wallet-file-secret \
  weblogic.domainUID=sample-domain1
    または、./kubernetes/samples/scripts/create-weblogic-domain/model-in-image/utils/opss-wallet.sh -rコマンドを使用して、ローカル・ウォレット・ファイルをシークレットとしてデプロイできます(このスクリプトのコマンドライン引数およびデフォルトを取得するには、-?を渡します)。
  • ドメインYAMLファイルのconfiguration.opss.walletPasswordSecretフィールドにOPSSパスワード・シークレットが指定されていることを確認し、ドメインYAMLファイルのconfiguration.opss.walletFileSecretフィールドにOPSSウォレット・ファイル・シークレットが指定されていることを確認します。
JRFドメインのデータベース・パスワードを変更する手順

JRFドメインがデータベース・パスワードの変更後もRCUデータに引き続きアクセスできるようにするには、次のステップに従います。

  • データベース・パスワードを変更する前に、データベース・スキーマにアクセスするすべてのドメインを停止します。 たとえば、serverStartPolicyをNEVERに設定します。

  • データベースのパスワードを更新します。

  • 各ドメインのRCUDbInfo.rcu_schema_passwordを含むKubernetesシークレットを更新します。

  • ドメインを再開します。 たとえば、serverStartPolicyをNEVERからIF_NEEDEDに変更します。

  • パスワードを変更すると別のウォレットが生成されるため、ウォレット・ファイルを再度保存します。
