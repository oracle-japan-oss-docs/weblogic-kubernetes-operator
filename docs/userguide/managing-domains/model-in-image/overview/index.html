導入

Model in Imageは、オペレータのDomain in ImageおよびDomain in PVドメイン・タイプにかわるものです。 オペレータ・ドメイン・タイプの比較については、「ドメイン・ホーム・ソース・タイプの選択」を参照してください。

Domain in PVおよびDomain in Imageとは異なり、Model in Imageでは、ドメインYAMLファイルをデプロイする前にWebLogicドメイン・ホームを事前作成する必要がありません。

次のことが可能です:

  • WebLogic Deploy Tool (WDT)モデル・ファイルおよびアプリケーション・アーカイブを使用したWebLogicドメイン・ホーム構成の定義。
  • モデル・ファイルおよびアーカイブをカスタム・コンテナ・イメージに埋め込み、WebLogic Image Tool (WIT)を使用してこのイメージを生成します。
  • Kubernetes ConfigMapを使用した追加モデル・ファイルの指定。
  • モデル内のマクロ参照を解決するKubernetesのシークレットの指定。 たとえば、シークレットを使用してデータベース資格証明を指定できます。
  • 実行時にWDTモデル・ファイルを更新しています。 たとえば、実行中のドメインにデータ・ソースを追加できます。 このような更新を有効にするには、現在すべての更新によってドメインがロールされることに注意してください。

この機能は、標準WLSドメイン、制限付きJRFドメインおよびJRFドメインでサポートされています。

JRFドメインの場合、Model in Imageでは、ドメインの初回起動時のインフラストラクチャ・データベースの初期化、データベース・パスワードの指定、および同じドメインの後続の再起動で再利用するためのデータベース・ウォレットの取得がサポートされます。 「JRFドメイン・タイプの要件」を参照してください。

WebLogic Deploy Toolモデル

WDTモデルは、WebLogic Scripting Tool (WLST)の構成スクリプトおよびテンプレートに代わる便利で単純なモデルです。 YAMLファイルを使用してWebLogicドメインをコンパクトに定義し、ZIPファイルにアプリケーション・アーカイブを含めることができます。 モデル形式およびModel in Imageとの統合の詳細は、「WebLogic Serverイメージ」および「モデル・ファイル」を参照してください。 WDTモデル形式は、オープン・ソースのWebLogic Deploy Tool GitHubプロジェクトで完全に記述されています。

実行時の動作

Model in ImageドメインYAMLファイルをデプロイする場合:

  • オペレータは、イントロスペクタ・ジョブと呼ばれる次のKubernetesジョブを実行します:

    • WDTアーティファクトをマージします。
    • WDTツールを実行してドメイン・ホームを生成します。
    • ドメイン・ホームをパッケージ化し、オペレータに渡します。

  • イントロスペクタ・ジョブが完了したら、次の手順を実行します:

    • オペレータは、DOMAIN_UID-weblogic-domain-introspect-cmという名前のConfigMapを作成し、その中にパッケージ化されたドメイン・ホームを配置します。
    • その後、オペレータはドメインのWebLogic Serverポッドをブートします。
    • ポッドは、ConfigMapからドメイン・ホームを取得します。

ランタイムの更新

モデルの更新は、初期デプロイメント後にイメージ、シークレットまたはWDTモデルのConfigMapを変更することで、実行時に適用できます。 イメージ名が変更されるか、ドメインrestartVersionが変更されると、イントロスペクタ・ジョブが再実行されて新しいドメイン・ホームが生成され、その後、変更されたドメイン・ホームがローリング・アップグレードを使用してドメインのWebLogic Serverポッドに伝播されます(ポッドごとに一度に再起動されます)。 「ランタイムの更新」を参照してください。

Model in ImageとCI/CDの連携の仕組みを理解するには、「CI/CDに関する考慮事項」を参照してください。
