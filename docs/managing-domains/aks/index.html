導入

Azure Kubernetes Service (AKS)を使用すると、Azureで管理されるKubernetesクラスタを簡単にデプロイできます。 AKSは、Kubernetesの管理の複雑さと運用オーバーヘッドを軽減するために、その責任の多くをAzureにオフロードします。 Azureは、ホスティングされたKubernetesサービスとして、ヘルス・モニタリングやメンテナンスなどの重要なタスクを処理します。 Kubernetesマスターは、Azureによって管理されます。 エージェント・ノードのみを管理および保守します。 管理対象Kubernetesサービスとして、AKSは無料です - マスターではなく、クラスタ内のエージェント・ノードに対してのみ支払います。

詳細は、「Azure Kubernetes Serviceとは」を参照してください

このドキュメントでは、Azure上でWebLogic Serverを簡単に開始できるAzure Marketplaceオファーについて説明します。 オファーは、すべての初期設定を処理し、AKSクラスタ、コンテナ・レジストリ、WebLogic Kubernetes Operatorインストールおよびモデル・イン・イメージ・ドメイン・ホーム・ソース・タイプを使用したドメイン作成を作成します。 ドメイン・ホームのソース・タイプの詳細は、「ドメイン・ホーム・ソース・タイプの選択」を参照してください。

Azure Marketplaceオファーを利用せずにWebLogic Kubernetes Operatorを手動で実行することもできます。 これを行うステップは、サンプルAzure Kubernetes Serviceに記載されています。

AKSクラスタの構成

「AKSクラスタの構成」ブレードを使用して、Oracle WebLogic ServerをAKSで実行する方法の基本詳細を構成します。 これを行うには、次の表に示すフィールドの値を入力します。

Azure Kubernetes Service

このセクションでは、WebLogic Serverを実行するAKSに関するいくつかのオプションを構成できます。

フィールド 説明
新しいAKSクラスタを作成 Yesに設定すると、デプロイメントによって、指定されたリソース・グループに新しいAKSクラスタ・リソースが作成されます。 Noに設定すると、デプロイメントを構成する既存のAKSクラスタを選択できます。 ノート: このオファーでは、既存のAKSクラスタにWebLogic関連のデプロイメントがないと想定しています。
サポートされている最新のAKS Kubernetesバージョンを使用 現在サポートされているバージョンは、1.24.3です。 Oracleは、「Azure Kubernetes Service (AKS)でサポートされているKubernetesバージョン」でAKSリリース・バージョンを追跡します。 新しいバージョンがリリースされると、Oracleはそのバージョンに対してAKS上のWLSを認定し、オファーをそのバージョンに更新します。 AKSを含む、Kubernetes上のWLSのOracleサポート・ポリシーについては、My Oracle SupportのWebLogic Kubernetes ToolKitサポート・ポリシー(ドキュメントID 2790123.1)およびAzureおよびOracle Linux (ドキュメントID 2914257.1)のサポート・ポリシーを参照してください。
ノード数 AKSクラスタ内のノードの初期数。 この値は、デプロイメント後に変更できます。 詳細については、「スケーリング」を参照してください。
ノード・サイズ デフォルトのVMサイズは、2x Standard DSv2、2 vcpus、7 GBのメモリーです。 別のVMサイズを選択する場合は、「サイズの変更」を選択し、VMサイズの選択ページでリストからサイズ(A3など)を選択し、「選択」を選択します。 仮想マシンのサイズ設定の詳細は、「サイズに関するAzureドキュメント」を参照してください。

Azureファイル共有サービスを使用した永続ボリュームの作成 選択すると、Azureストレージ・アカウントおよびAzureファイルの共有がプロビジョニングされます。Azureファイル共有の静的永続ボリュームが、AKSクラスタのノードにマウントされます。 詳細は、「Oracle WebLogic Server永続ストレージ」および「AKSでAzure Files共有の永続ボリューム」を参照してください。

イメージの選択

この項では、モデル・イン・イメージ・ドメインのホーム・ソース・タイプを使用してデプロイされるイメージを構成できます。 WebLogicイメージにはいくつかのオプションがあり、そこにデプロイされているアプリケーション・イメージがあります。

フィールド 説明
Oracle Container Registry (OCR)から既存のWebLogic Server Dockerイメージを使用 Yesに設定すると、後続のオプションが制約され、Oracle Container Registryに格納されている既存のWebLogic Server Dockerイメージ・セットからのみ選択できるようになります。 Noに設定すると、ユーザーは既存のAzureコンテナ・レジストリを参照し、ドメインの作成に使用されるそのレジストリ内のWebLogic ServerイメージのDockerタグを指定する必要があります。 指定されたイメージは、WebLogic Kubernetes Operatorと互換性があるとみなされます。 これにより、特定の個別パッチまたはOracle四半期パッチ(PSU)を含むイメージなどのカスタム・イメージを使用できます。 WebLogic Serverイメージの詳細は、「WebLogicイメージ」を参照してください。
アプリケーション・イメージを格納する新しいAzureコンテナ・レジストリを作成 Yesに設定すると、デプロイメントで使用するイメージを保持する新しいAzureコンテナ・レジストリ(ACR)が作成されます。 Noに設定した場合は、既存のACRを指定する必要があります。 この場合、選択したACRの管理アカウントが有効になっていることを確認する必要があります。 詳細については、「管理アカウント」を参照してください。
既存のACRインスタンスの選択 このオプションは、「Oracle Container Registryから既存のWebLogic Server Dockerイメージを使用」がNoに設定されている場合にのみ表示されます。 表示されている場合は、既存のAcure Container Registryインスタンスを選択します。
イメージ・パスを指定してください このオプションは、「Oracle Container Registryから既存のWebLogic Server Dockerイメージを使用」がNoに設定されている場合にのみ表示されます。 表示される場合、値は、指定されたACR内のイメージの完全修飾Dockerタグである必要があります。
Oracle Single Sign-On認証のユーザー名 選択したWebLogic Serverイメージの使用条件および制限が受け入れられたOracleシングル・サインオン・アカウントのユーザー名。
Oracle Single Sign-On認証のパスワード そのアカウントのパスワード。
パスワードの確認 前のフィールドの値を再入力します。
指定されたSSOアカウントはアクティブなOracleサポート契約に関連付け Yesに設定されている場合、middleware/weblogic_cpuリポジトリでライセンス契約を受け入れる必要があります。 Noに設定した場合は、middleware/weblogicでライセンス契約に同意する必要があります。 ライセンス契約を受け入れるステップ: Oracle Container Registryにログインします。middleware/weblogic_cpuおよびmiddleware/weblogicリポジトリに移動し、ライセンス契約に同意します。 詳細は、この「ドキュメント」を参照してください。
WebLogic Server Dockerタグの選択 サポートされているイメージのいずれかを選択します。

Java EEアプリケーション

この項では、Java EEアプリケーションをWebLogic Serverデプロイメントとともにデプロイできます。

フィールド 説明
アプリケーション・パッケージをデプロイ Yesに設定されている場合、Java EE WAR、EARまたはJARファイルを、選択したバージョンのWebLogic Serverでのデプロイメントに適したファイルを指定する必要があります。 Noに設定すると、アプリケーションはデプロイされません。
アプリケーション・パッケージ(.war、.ear、.jar) 「参照」ボタンを使用すると、そのアカウント内の既存のAzureストレージ・アカウントおよびストレージ・コンテナからファイルを選択できます。 ストレージ・アカウントおよびコンテナの作成方法については、「ストレージ・アカウントの作成」および「ストレージ・コンテナの作成とアプリケーション・ファイルのアップロード」を参照してください。
アプリケーションがACTIVEになっていない場合、デプロイメントに失敗しました。 選択すると、デプロイメントは、デプロイされたアプリケーションがACTIVE状態に達するまで待機し、ない場合はデプロイメントに失敗します。 詳細については、「Oracleのドキュメント」を参照してください。
WebLogic管理対象サーバー・レプリカの数 ドメインのreplicasフィールドの初期値。 詳細については、「スケーリング」を参照してください。

選択内容に問題がなければ、「次 : TLS/SSL構成」を選択します。

TLS/SSL構成

「TLS/SSL構成」ブレードでは、認証局(CA)によって提供される独自のSSL証明書を使用して、セキュアなHTTPSポートでOracle WebLogic Server管理コンソールを構成できます。 詳細は、「Oracle WebLogic Serverキーストアの構成」を参照してください。

プリファレンスに基づいてオプション「独自のTLS/SSL証明書を使用してHTTPS (セキュア)ポートを使用するように、WebLogic Server管理コンソール、リモート・コンソールおよびクラスタを構成します。」に対してYesまたはNoを選択します。 Noを選択した場合は、詳細を指定する必要はなく、「次 : ネットワーク」を選択して続行できます。 Yesを選択した場合、既存のキーストアをアップロードするか、Azure Key Vaultに格納されているキーストアを使用するかを選択して、必要な構成の詳細を指定できます。

既存のキーストアをアップロードする場合は、オプション「必要な構成をどのように提供しますか」の「既存のKeyStoresのアップロード」を選択し、次の表に示すフィールドの値を入力します。

Upload existing KeyStores

フィールド 説明
アイデンティティKeyStoreデータfile(.jks,.p12) 次の手順を実行して、カスタムIDキーストアのデータ・ファイルをアップロードします:
1 ファイルのアイコンをクリックします。
2 IDキーストア・ファイルが存在するフォルダに移動し、ファイルを選択します。
3 「開く」をクリックします。
パスワード カスタムIDキーストアのパスフレーズを入力します。

有用なリソース

次の有用なリソースを確認します。

TLS/SSL証明書をキー・ボールトに格納

  1. Base 64では、証明書ファイルをエンコードします。macOSの-w0は省略します:

    base64 myIdentity.jks -w0 >mycert.txt
# base64 myIdentity.p12 -w0 >mycert.txt
# base64 myTrust.jks -w0 >mycert.txt
# base64 myTrust.p12 -w0 >mycert.txt
# base64 root.cert -w0 >mycert.txt
# base64 gatewayCert.pfx -w0 >mycert.txt

  2. Azureポータルから、キー・ボールトを開きます。

  3. 「設定」セクションで、「シークレット」を選択します。

  4. 「生成/インポート」を選択します。

  5. 「アップロード」オプション。デフォルト値はそのままにします。

  6. 名前に、myIdentityCertDataまたは任意の名前を入力します。

  7. 値に、mycert.txtファイルの内容を入力します。

  8. 残りの値はデフォルトのままにし、「作成」を選択します。
