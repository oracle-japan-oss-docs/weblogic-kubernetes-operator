WDTプロジェクト・リリース・プロセス

このドキュメントでは、WebLogic Deploy Tooling (WDT)リリースを作成するために従う必要があるプロセスについて説明します。

前提条件

  • WebLogic Server 12.2.1.xのローカル・インストールが使用可能である必要があります。
  • リリース・プロセスはマスター・ブランチに直接プッシュするため、リリース・プロセスを実行するユーザーには「WebLogic Deploy Tooling GitHubリポジトリ」に対する管理権限が必要です。
  • リリース・プロセスを実行するユーザーは、(少なくとも) repo:status, repo_deployment, public_repoおよびsecurity_events権限を持つリポジトリのGitHub個人アクセス・トークンを作成する必要があります。
  • リリースを実行するユーザーは、次に示すように、GitHub個人アクセス・トークンが格納されているMaven settings.xmlファイルにサーバーを追加する必要があります。 このトークンは、プレーン・テキストで格納することも、「Mavenパスワードの暗号化」を使用して暗号化することもできます。 
  <servers>
    <server>
      <id>github</id>
      <passphrase>store plain text or encrypted token here</passphrase>
    </server>
  </servers>

ソフトウェア・リリース・プロセス

ベスト・プラクティスは、次のステップを開始する前に、GitHubに公開されるリリース・ノートを記述することです。

  1. 環境変数WLST_DIRを<WLS-install-dir>/oracle_common/common/binに設定(およびエクスポート)し、<WLS-install-dir>をWLS 12.2.1.xインストール・ディレクトリのフルパスに置き換えます。
  2. weblogic-deploy-toolingプロジェクト・ディレクトリで、次の例のような内容のrelease.propertiesというファイルを作成します。 この例は、1.9.11リリースを切り取るように構成されていることに注意してください。 
tag=release-1.9.11
releaseVersion=1.9.11
developmentVersion=1.9.12-SNAPSHOT
  1. weblogic-deploy-toolingプロジェクト・ディレクトリで、mvn -B -P release release:prepare release:performコマンドを実行します。 SSH秘密キーにパスフレーズがある場合は、パスフレーズを複数回要求するため、ビルドをよく確認してください。 適時に入力しないと、失敗する可能性があります。
  2. ビルドが失敗した場合は、問題を修正した後に、mvn -B -P release release:rollbackコマンドを実行してそれを元に戻し、ステップ2からやり直します。
  3. ソフトウェアがリリースされたら、GitHubリリース・プロセスに進みます。

GitHubリリース・プロセス

このプロセスは、ローカルのMavenリポジトリにあるWDTインストーラに依存することに注意してください。 したがって、同じユーザーが前の項のステップと同じマシンでこれらのステップを実行することが重要です。

  1. リリース・ノートを<wdt-project-directory>/target/ReleaseNotes.mdファイルに保存します。
  2. コマンドmvn -f github-release.xml -DreleaseVersion=<release version number> verifyを実行して、ドラフトのGitHubリリースを作成します。
  3. GitHubにログインし、Releasesページに移動し、ドラフト・リリースを確認および編集してから、リリースを公開します。
